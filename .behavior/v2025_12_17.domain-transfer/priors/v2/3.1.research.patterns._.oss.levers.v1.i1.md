# research.patterns: OSS levers for declastruct-cloudflare

## overview

this document evaluates open source tools available to fulfill the declastruct-cloudflare vision for declarative domain management.

---

## lever.1: cloudflare official typescript sdk

**package**: `cloudflare` on npm [1]

**version**: 5.2.0 (current in package.json)

**what**:
> "The official TypeScript library for the Cloudflare API" [1]

**features** [1]:
- zones: create, list, get, update, delete
- dns records: full CRUD operations
- registrar domains: list, get, update
- intel WHOIS: domain lookup for transfer detection

**usage** [2]:
```typescript
import Cloudflare from 'cloudflare';
const client = new Cloudflare({ apiToken: process.env['CLOUDFLARE_API_TOKEN'] });

// dns record creation
const recordResponse = await client.dns.records.create({
  zone_id: '023e105f4ecef8ad9ca31a8372d0c353',
  name: 'example.com',
  ttl: 3600,
  type: 'A',
});
```

**maintenance** [1]:
- maintained by cloudflare directly
- active development (v4.4.1+ includes simplified worker deployment)

**pros**:
- official sdk with typescript types
- complete api coverage
- auto-pagination support
- supports zones, dns, registrar, whois

**cons**:
- imperative api (not declarative)
- requires dao wrapper for declastruct integration

**verdict**: [USE] — primary lever for cloudflare api access

## lever.2: declastruct framework

**package**: `declastruct` on npm [3]

**version**: 1.7.3 (current in package.json)

**what**:
> "declarative control of any resource constructs, batteries included" [3]

**features** [3]:
- stateless comparison against live remote state
- plan/apply workflow
- works with any api
- enforces idempotency via dao interfaces

**usage** [3]:
```typescript
// declare desired state
export const getResources = async () => {
  const bucket = DeclaredAwsS3Bucket.as({
    name: 'your-s3-bucket',
    version: true,
  });
  return [bucket];
};

// plan changes
// npx declastruct plan --wish resources.ts --into plan.json

// apply changes
// npx declastruct apply --plan plan.json
```

**key benefit** [3]:
> "no state files to manage — compares directly against live remote state"

**dao interface contract** [3]:
- `getOne({ by: { primary | unique | ref } })`
- `getAll()`
- `set({ findsert | upsert })`
- `del({ by })`

**maintenance**:
- active development by ehmpathy

**pros**:
- stateless (no state files)
- typescript-native
- composable with domain-objects
- enforces pit-of-success patterns

**cons**:
- requires provider implementation per platform

**verdict**: [USE] — core framework for declarative control

## lever.3: domain-objects library

**package**: `domain-objects` on npm [4]

**version**: 0.31.7 (current in package.json)

**what**:
> "simple, convenient way to represent domain objects, leverage domain knowledge, and add runtime validation" [4]

**features** [4]:
- DomainEntity for identity-based objects
- DomainLiteral for value objects
- static primary, unique, metadata declarations
- runtime validation support
- serialize/deserialize utilities

**usage** [4]:
```typescript
import { DomainEntity } from 'domain-objects';

interface DeclaredCloudflareDomainZone {
  id?: string;          // @metadata
  name: string;         // @unique
  type: 'full' | 'partial';
}

class DeclaredCloudflareDomainZone
  extends DomainEntity<DeclaredCloudflareDomainZone>
  implements DeclaredCloudflareDomainZone
{
  public static primary = ['id'] as const;
  public static unique = ['name'] as const;
  public static metadata = ['id'] as const;
}
```

**key benefit** [4]:
> "Domain models inform us of what properties uniquely identify a domain object"

**maintenance**:
- active development by ehmpathy

**pros**:
- typed entity declarations
- runtime validation
- identity semantics (primary, unique keys)
- reference types (RefByPrimary, RefByUnique)

**cons**:
- curve to learn static property conventions

**verdict**: [USE] — domain object model foundation

## lever.4: type-fns utility library

**package**: `type-fns` on npm [5]

**version**: 1.21.0 (current in package.json)

**what**:
> "collection of generic types, type guards, and type checks" [5]

**key utility** [5]:
- PickOne: ensures exactly one key in object is defined

**usage** [5]:
```typescript
import { PickOne } from 'type-fns';

const setDomainZone = async (
  input: PickOne<{
    findsert: DeclaredCloudflareDomainZone;
    upsert: DeclaredCloudflareDomainZone;
  }>,
  context: ContextCloudflareApi,
) => { ... };
```

**other utilities** [5]:
- isPresent: type guard for null/undefined filter
- isOfEnum: enum validation
- HasMetadata: type for metadata-populated objects

**maintenance**:
- active development by ehmpathy

**pros**:
- PickOne enables findsert/upsert pattern
- type guards simplify runtime checks
- composable with domain-objects

**cons**:
- none significant

**verdict**: [USE] — essential for idempotent operation signatures

## lever.5: helpful-errors library

**package**: `helpful-errors` on npm [6]

**version**: 1.5.3 (current in package.json)

**what**:
> "Standardized helpful errors and methods for simpler, safer, and easier to read code" [6]

**error types** [6]:
- UnexpectedCodePathError: for impossible states
- BadRequestError: for invalid caller inputs
- HelpfulError: base class for custom errors

**usage** [6]:
```typescript
import { BadRequestError, UnexpectedCodePathError } from 'helpful-errors';

// invalid request
if (!user)
  throw new BadRequestError('user does not exist for uuid', { userUuid });

// unexpected state
if (lowestTirePressure === undefined)
  throw new UnexpectedCodePathError('no tire pressures found', { tires });
```

**key benefit** [6]:
> "makes it very easy to specify what the issue was as well as any other information that may be helpful"

**transfer guidance pattern**:
```typescript
BadRequestError.throw(
  `domain "mhember.com" not found in cloudflare registrar

transfer required
  ├─ at godaddy (current registrar)
  │  └─ steps: unlock, get auth code
  └─ at cloudflare
     └─ steps: enter code, confirm`,
  { domain, registrar, accountId },
);
```

**maintenance**:
- active development by ehmpathy

**pros**:
- structured error messages with metadata
- distinguishes bad-request vs code-error
- enables guided operations pattern

**cons**:
- none significant

**verdict**: [USE] — essential for transfer/purchase guidance

## lever.6: terraform cloudflare provider

**package**: `cloudflare/cloudflare` on terraform registry [7]

**what**:
> "manage the Cloudflare global network with the same familiar tools you use to automate the rest of your infrastructure" [7]

**features** [7]:
- zone management
- dns record CRUD
- zone settings, rate limits, load balance

**usage** [8]:
```hcl
resource "cloudflare_zone" "example" {
  zone = "example.com"
}

resource "cloudflare_dns_record" "example" {
  zone_id = cloudflare_zone.example.id
  name    = "api"
  content = "192.168.1.1"
  type    = "A"
  ttl     = 300
}
```

**key benefit** [7]:
> "Automation – Define DNS configurations once and apply them consistently"

**maintenance**:
- maintained by cloudflare

**pros**:
- mature infrastructure-as-code solution
- state management built-in
- large community

**cons**:
- requires HCL (not typescript)
- state files to manage
- not composable with application code

**verdict**: [SKIP] — declastruct provides typescript-native alternative

## lever.7: pulumi cloudflare provider

**package**: `@pulumi/cloudflare` [9]

**what**:
> "build, deploy, and manage Cloudflare resources with standard languages" [9]

**features** [9]:
- workers, routes, dns records
- zones and zone settings
- typescript/javascript support

**key benefit** [10]:
> "you get familiar constructs like conditionals, loops, functions, and classes" [10]

**maintenance**:
- maintained by pulumi
- uses terraform provider under the hood [9]

**pros**:
- typescript support
- full language features

**cons**:
- separate infrastructure tool
- not integrated with domain-objects
- state management required

**verdict**: [SKIP] — declastruct provides better integration with ehmpathy patterns

## lever.8: test-fns library

**package**: `test-fns` on npm [11]

**version**: 1.7.2 (current in package.json)

**what**:
> "write usecase-driven tests for simpler, safer code" [11]

**features** [11]:
- given/when/then BDD structure
- useBeforeAll for shared setup
- runIf/skipIf for conditional tests

**usage** [11]:
```typescript
import { given, when, then, useBeforeAll } from 'test-fns';

describe('doesPlantNeedWater', () => {
  given('a plant', () => {
    when('the plant doesnt have enough water', () => {
      const plant: Plant = { id: 7, hydration: 'DRY' };
      then('it should return true', () => {
        expect(doesPlantNeedWater(plant)).toEqual(true);
      });
    });
  });
});
```

**maintenance**:
- active development by ehmpathy

**pros**:
- BDD structure improves readability
- useBeforeAll reduces setup duplication
- composable test patterns

**cons**:
- none significant

**verdict**: [USE] — established test pattern in codebase

## recommended stack

**production code**:
1. `cloudflare` sdk — api access layer
2. `declastruct` — declarative framework
3. `domain-objects` — entity model
4. `type-fns` — PickOne for findsert/upsert
5. `helpful-errors` — guidance for transfers/purchases

**test code**:
1. `test-fns` — BDD test structure
2. `jest` — test runner
