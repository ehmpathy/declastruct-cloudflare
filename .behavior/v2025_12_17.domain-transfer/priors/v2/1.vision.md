## the outcome world

### day-in-the-life

**am standup**: "we need to spin up infrastructure for mhember.com — dns records for api, www, mail, plus transfer the domain from godaddy."

**with declastruct-cloudflare**:

```ts
// resources.ts
import { refByUnique } from 'domain-objects';

export const getResources = async () => {
  // declare the zone
  const zone = new DeclaredCloudflareDomainZone({
    name: 'mhember.com',
    type: 'full',
  });

  // declare dns records — use refByUnique to extract ref
  const apiRecord = new DeclaredCloudflareDomainDnsRecord({
    zone: refByUnique(zone),
    name: 'api.mhember.com',
    type: 'A',
    content: '192.168.1.100',
    ttl: 300,
    proxied: true,
  });

  const wwwRecord = new DeclaredCloudflareDomainDnsRecord({
    zone: refByUnique(zone),
    name: 'www.mhember.com',
    type: 'CNAME',
    content: 'mhember.com',
    ttl: 1,  // automatic
    proxied: true,
  });

  const mxRecord = new DeclaredCloudflareDomainDnsRecord({
    zone: refByUnique(zone),
    name: 'mhember.com',
    type: 'MX',
    content: 'mail.example.com',
    ttl: 3600,
    priority: 10,
  });

  // declare domain registration settings
  const registration = new DeclaredCloudflareDomainRegistration({
    name: 'mhember.com',
    zone: refByUnique(zone),
    autoRenew: true,
    locked: true,
    privacyProtection: true,
  });

  return [zone, apiRecord, wwwRecord, mxRecord, registration];
};
```

**run plan**:
```sh
$ declastruct plan --wish resources.ts --into plan.json
```

declastruct generates `plan.json` with a `changes` array that declares what will happen.

**run apply**:
```sh
$ declastruct apply --plan plan.json
```

for the registration, `setDomainRegistration` throws a **BadRequestError** with treestruct guidance:

```ts
BadRequestError.throw(
  `domain "mhember.com" not found in cloudflare registrar

⚠️ transfer required
   │
   ├─ at godaddy (current registrar)
   │  ├─ refs
   │  │  ├─ https://developers.cloudflare.com/registrar/get-started/transfer-domain-to-cloudflare/
   │  │  └─ https://www.godaddy.com/help/transfer-my-domain-away-from-godaddy-3560
   │  └─ steps
   │     ├─ 1. disable dnssec
   │     ├─ 2. disable whois privacy
   │     ├─ 3. unlock domain
   │     └─ 4. get auth code
   │
   ├─ at cloudflare
   │  ├─ refs
   │  │  └─ https://dash.cloudflare.com/${accountId}/domains/transfer
   │  └─ steps
   │     ├─ 5. enter domain + auth code
   │     ├─ 6. confirm contact
   │     └─ 7. complete transfer
   │
   └─ post-initiation
      ├─ 8. approve FOA email
      ├─ 9. wait up to 5 days
      └─ 10. re-run declastruct apply`,
  { domain: 'mhember.com', registrar: 'godaddy', accountId },
);
```

**purchase guidance** (when domain is unregistered):

```ts
BadRequestError.throw(
  `domain "mhember.com" not found in cloudflare registrar

⚠️ purchase required
   │
   ├─ domain is available for registration
   │
   └─ at cloudflare
      ├─ refs
      │  └─ https://dash.cloudflare.com/${accountId}/domains/register
      └─ steps
         ├─ 1. search for "${domain}"
         ├─ 2. add to cart
         ├─ 3. complete purchase
         └─ 4. re-run declastruct apply`,
  { domain: 'mhember.com', accountId },
);
```

**how declastruct detects transfer vs purchase**:

declastruct uses cloudflare's WHOIS API (`/accounts/{account_id}/intel/whois?domain=`) to determine if a domain is registered elsewhere:

1. `setDomainRegistration({ findsert })` is called for a domain
2. check if domain exists in cloudflare registrar → if yes, return it
3. if not, call WHOIS API to check if domain is registered elsewhere
4. if WHOIS shows registration → throw **transfer guidance** HelpfulError
5. if WHOIS shows unregistered → throw **purchase guidance** HelpfulError

this allows declastruct to give the right guidance automatically — no separate "transfer" resource needed. the user declares the desired end state, declastruct figures out the path.

**additional guidance enhancements**:
- WHOIS response includes `current_registrar` — we can say "at godaddy" instead of "at current registrar"
- registrar-specific transfer guides:
  - godaddy: https://www.godaddy.com/help/transfer-my-domain-away-from-godaddy-3560
  - squarespace: https://support.squarespace.com/hc/en-us/articles/205812378-Transferring-a-domain-away-from-Squarespace
  - aws route53: https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/domain-transfer-from-route-53.html
- `transferIn` status fields track progress — guidance can resume where you left off
- cloudflare API includes `nameservers` — we can show exact NS records to update

**after manual steps + re-apply**: the registration findsert succeeds, settings are applied

**end of day**: infrastructure is declared in code, versioned in git, and reproducible.

### the "aha" moment

the aha moment comes when you realize:

1. **you can declare dns records for zones that don't exist yet** — the `zone: { name: 'mhember.com' }` reference is resolved at apply time. declare your intent, let declastruct sort out the order.

2. **drift detection is automatic** — run `declastruct plan` against prod to see what's changed. someone added a record via dashboard? plan shows it.

3. **domain transfers aren't scary anymore** — declastruct guides you through every step with clear instructions and links. HelpfulError provides the exact steps at both registrars.

### contract inputs & outputs

**provider setup**:
```ts
const provider = await getDeclastructCloudflareProvider({
  apiToken: process.env.CLOUDFLARE_API_TOKEN,
  accountId: process.env.CLOUDFLARE_ACCOUNT_ID,
}, { log: console });
```

**domain objects**:
- `DeclaredCloudflareDomainZone` — container for dns, gateway to registrar
- `DeclaredCloudflareDomainDnsRecord` — individual dns records (A, AAAA, CNAME, MX, TXT, etc)
- `DeclaredCloudflareDomainRegistration` — registrar-level domain management

**references**:

devtime references use `RefByUnique` — intellisense shows exactly what properties are needed:

```ts
// in domain object definition
interface DeclaredCloudflareDomainDnsRecord {
  zone: RefByUnique<typeof DeclaredCloudflareDomainZone>;  // { name: string }
  // ...
}

// pass instance directly — shape fits RefByUnique
new DeclaredCloudflareDomainDnsRecord({
  zone: refByUnique(zone),
  // ...
});
```

runtime references use `refByUnique` to extract refs from instances:

```ts
import { refByUnique, RefByUnique } from 'domain-objects';

// have instance — use refByUnique(instance)
const zoneRef = refByUnique(zone);
// zoneRef = { name: 'mhember.com' }

// no instance — use RefByUnique.as<typeof Class>({ ... })
const zoneRef = RefByUnique.as<typeof DeclaredCloudflareDomainZone>({ name: 'mhember.com' });
```

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| see what domains are in cloudflare | ✅ | `getAllDomainZones()`, `getAllDomainRegistrations()` |
| register new domains | ⚠️ guided | api doesn't support; HelpfulError provides step-by-step |
| manage dns of those domains | ✅ | full CRUD via `DomainDnsRecord` operations |
| import domains from another registrar | ⚠️ guided | api doesn't support; HelpfulError provides step-by-step |
| idempotent operations | ✅ | findsert/upsert semantics throughout |
| drift detection | ✅ | `declastruct plan` compares declared vs actual |
| audit trail | ✅ | resources.ts in git |

### pros

1. **declarative** — describe desired state, not imperative steps
2. **idempotent** — safe to run multiple times
3. **typed** — typescript catches errors at compile time
4. **guided** — when automation isn't possible, guidance is clear
5. **composable** — zones, records, registrations work together
6. **observable** — plan shows exactly what will change

### cons

1. **domain registration/transfer requires manual steps** — cloudflare api limitation, not declastruct limitation
2. **learn curve** — need to understand zone vs registration concepts
3. **eventual consistency** — dns propagation takes time regardless of tools

### edge cases and pit of success

| edge case | how declastruct keeps you safe |
|-----------|-------------------------------|
| declare dns record before zone exists | ✅ zone ref resolved at apply time |
| try to delete zone with records | ✅ declastruct orders operations correctly |
| domain transfer fails midway | ✅ `transferIn` status tracked; guidance resumes where you left off |
| zone awaits nameserver change | ✅ HelpfulError guides you to update nameservers at registrar |
| A record + CNAME on same name | ✅ cloudflare api rejects; error bubbles up |
| typo in domain name | ⚠️ caught at plan/apply time (cloudflare validates) |
| wrong account id | ✅ fail-fast with clear BadRequestError |
| expired api token | ✅ fail-fast with authentication error |

### what's awkward?

1. **domain registration is "guided" not "automated"** — users might expect full automation, but cloudflare's api doesn't support it. the HelpfulError pattern makes this explicit.

2. **zone vs registration distinction** — users think "domain" but need to understand zone (dns container) vs registration (ownership record). documentation and name choices help clarify.

3. **transfer wait period** — 5-day icann wait period is unavoidable. declastruct tracks status but can't speed up the process.

## domain knowledge distillation

as we implement this, we should distill key lessons about cloudflare's domain model into `.agent/repo=.this/role=any/briefs/` so future agents (and humans) can leverage this knowledge.

### briefs to create

| brief | content |

| `ref.cloudflare.zone-activation.md` | awaited status, nameserver delegation, propagation duration |

### key domain lessons to capture

1. **zone ≠ registration** — zone is dns container, registration is ownership. both may exist independently.

2. **full setup required for registrar** — domains that use cloudflare registrar must have cloudflare as authoritative dns (zone type = "full")

3. **transfer has icann-mandated wait** — 5-day window is unavoidable, not a cloudflare limitation

4. **60-day lock after registration/transfer** — domains cannot be transferred within 60 days of registration or previous transfer

5. **whois api for detection** — `/intel/whois` endpoint reveals current registrar, to enable smart guidance

6. **transferIn status is trackable** — `accept_foa`, `enter_auth_code`, `unlock_domain` fields show transfer progress

7. **dns record unique key is composite** — `(zone_id, name, type)` — same name can have multiple record types

8. **ttl varies by plan** — minimum 60s for free/pro/business, 30s for enterprise

## summary

declastruct-cloudflare transforms cloudflare domain management from click-ops to code-ops. declare your zones, dns records, and registration settings in typescript. let declastruct handle the diff, the order of operations, and the edge cases. when cloudflare's api can't automate one task, declastruct guides you through it with clear, actionable steps.

**the world with this wish fulfilled**: domain infrastructure is versioned, reproducible, and reviewable — just like application code.
