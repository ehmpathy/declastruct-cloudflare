# research.claims: declastruct-cloudflare domain management

## claims discovered

### dns & infrastructure-as-code

**[FACT]** dns record management is where most teams start with infrastructure as code [1]
> "DNS record management is where most teams start with Infrastructure as Code."

**[FACT]** declarative configuration focuses on desired state, not step-by-step instructions [1]
> "A declarative configuration is an approach to defining and managing the desired state of a system or application without specifying the step-by-step instructions for achieving that state."

**[OPIN]** if it affects production traffic, it should live in git [1]
> "If it affects production traffic, it should live in Git. Managing it outside version control increases operational risk."

**[KHUE]** most infrastructure teams still handle domain registrations through manual processes — creates disconnect between dns records (versioned) and domain configurations (manual) [1]
> "Most infrastructure teams managing DNS still handle domain registrations through manual processes or separate systems. This creates a disconnect: your DNS records are in version control, but your domain configurations, renewal settings, and security features are managed elsewhere."

### cloudflare zone management

**[FACT]** cloudflare zones have six statuses: initializing, pending, active, moved, deleted, purged [2]
> "Pending zones cannot proxy traffic through Cloudflare. They remain in limbo until activation or removal occurs."

**[FACT]** zones require nameserver update (full setup) or txt record verification (partial setup) to activate [2]
> "Full Setup: Update your domain's authoritative nameservers to Cloudflare's assigned servers"

**[FACT]** free plan zones auto-delete after 28 days if unactivated [2]
> "Free plans auto-delete after 28 days if unactivated; paid plans remain pending indefinitely until activated or removed."

**[SUMP]** users expect zones to work immediately after creation — they may not understand queued status requires manual nameserver change [2]

### icann transfer requirements

**[FACT]** icann prohibits domain transfers within 60 days of initial registration [3]
> "ICANN's transfer policy prohibits inter-registrar transfers within 60 days of initial registration. This mandatory lock cannot be waived or opted out."

**[FACT]** icann prohibits domain transfers within 60 days of registrant contact change [3]
> "After 1 December 2016, registrars must impose a lock that will prevent any transfer to another registrar for sixty (60) days following a change to a registrant's information."

**[FACT]** registrars may allow opt-out of 60-day lock for registrant changes (but not for initial registration) [3]
> "registrars may (but are not required to) allow registrants to opt out of the 60-day lock prior to the change of registrant request."

**[FACT]** the 60-day lock is a security measure to prevent domain theft [3]
> "The 60-day transfer lock after a registrant change is an ICANN security measure designed to prevent domain hijacking."

**[FACT]** icann does not manage locks directly; registrars are responsible [3]
> "ICANN does not place, nor can it lift a 60-day lock on domain names. The registrars are responsible for imposing and managing these locks."

### cloudflare-specific requirements

**[FACT]** cloudflare registrar only supports transfers for domains active on cloudflare full setup [4]
> "Cloudflare Registrar only supports transfers of domains that are active on a Cloudflare full setup"

**[FACT]** authorization codes expire within 5-14 days [4]
> "Some authorization codes expire within 5-14 days"

**[FACT]** each transfer adds one year to domain registration [4]
> "Each transfer adds one year to your domain registration"

**[FACT]** domain cannot be within 15 days of expiration to transfer [4]

---

### idempotency patterns

**[FACT]** upsert pattern = update if extant, insert if absent — fundamental idempotent operation [5]
> "A common example of an idempotent operation is an upsert (update or insert) statement in a database. If the statement is run multiple times with the same data, the result will be the same as if it were run only once."

**[FACT]** unique identifiers (idempotency keys) prevent duplicate mutations [5]
> "use Unique Identifiers by attaching a unique ID (idempotency key) to each request to track and prevent duplicate processing"

**[OPIN]** some dns operations are idempotent by nature (e.g., upsert a record), others require idempotency keys [5]
> "Some foreign state mutations are idempotent by nature (e.g. adding a DNS record), while some are not idempotent but can be made idempotent with the help of an idempotency key."

**[SUMP]** findsert (find-or-insert) is safer than upsert for cases where unintended overwrites could break production — declastruct uses both patterns with explicit mode selection

---

### dns record constraints

**[FACT]** A/AAAA records cannot coexist with CNAME records on the same name [6]
> "A/AAAA records cannot coexist with CNAME records on the same name"

**[FACT]** NS records cannot share names with other record types [6]
> "NS records cannot share names with other record types"

**[FACT]** TTL minimum is 60s for most plans, 30s for enterprise [6]
> "TTL values must fall between 60 and 86,400 seconds... Enterprise zones support minimum values as low as 30 seconds."

## open questions

**[KHUE]** how should declastruct handle queued zone status? options:
- throw HelpfulError with nameserver update guidance
- return zone and let user manage activation separately
- poll until active (could take 24+ hours)

**[KHUE]** should drift detection include orphan detection (records created via dashboard that aren't in code)? requires state file or alternative mechanism.

**[KHUE]** how should rate limits be handled? cloudflare api has undocumented rate limits that could affect bulk operations.
