# self-review: has-questioned-5whys

## decision.1: add WHOIS lookup for transfer/purchase detection

### 5 whys

1. **why add WHOIS lookup?**
   → because setDomainRegistration needs to know if domain is registered elsewhere or unregistered

2. **why does it need to know this?**
   → because the guidance message differs: "transfer from godaddy" vs "purchase available"

3. **why does guidance need to differ?**
   → because transfer and purchase are fundamentally different user journeys with different steps

4. **why are they different journeys?**
   → because transfer requires coordination with current registrar (unlock, auth code) while purchase is single-registrar

5. **why can't we automate both?**
   → because cloudflare API does not support transfers or purchases programmatically

**root reason**: cloudflare's API limitation forces manual steps, so we provide accurate guidance based on domain status

---

## decision.2: separate domainWhoisRecord/ directory

### 5 whys

1. **why create a separate directory?**
   → because WHOIS lookup is a distinct operation from registration management

2. **why does it need to be distinct?**
   → because the cast function requires unit tests, which need isolation

3. **why does the cast function need unit tests?**
   → because it transforms external API response into typed domain object

4. **why not inline the transform?**
   → because inline transforms are harder to test without mocks

5. **why avoid mocks?**
   → because mocks hide real bugs and create false confidence (rule.forbid.remote-boundaries)

**root reason**: testability via cast function extraction aligns with codebase test patterns

---

## decision.3: throw BadRequestError instead of UnexpectedCodePathError

### 5 whys

1. **why throw BadRequestError for guidance?**
   → because "domain not found" is not a bug, it's a user action required

2. **why does the error type matter?**
   → because different error types signal different failure modes

3. **why signal different failure modes?**
   → because users and systems respond differently to bugs vs bad requests

4. **why do they respond differently?**
   → because UnexpectedCodePathError suggests retry won't help (bug), BadRequestError suggests user can fix

5. **why is this distinction important?**
   → because correct error semantics enable correct monitor alerts and UX

**root reason**: error semantics communicate intent — BadRequestError says "you can fix this"

---

## decision.4: 4 fields in DeclaredCloudflareDomainWhoisRecord

### 5 whys

1. **why only 4 fields (domain, found, registrar, registrarName)?**
   → because guidance logic only uses these fields

2. **why does guidance only use these fields?**
   → because transfer/purchase detection needs: is it registered? where?

3. **why not add more fields for future use?**
   → because YAGNI — we add complexity when needed, not before

4. **why is YAGNI important here?**
   → because unused fields create maintenance burden and deceptive contracts

5. **why avoid deceptive contracts?**
   → because consumers may rely on fields we never populate correctly

**root reason**: minimal interface reduces surface area for bugs and confusion

---

## decision.5: WHOIS is internal, not exported in SDK

### 5 whys

1. **why not export getOneDomainWhoisRecord publicly?**
   → because users never call WHOIS lookup directly

2. **why don't users call it directly?**
   → because the user journey is: declare registration → get guidance → follow steps

3. **why is that the user journey?**
   → because declastruct is declarative — users declare desired state, not imperatively query

4. **why not support imperative queries too?**
   → because more exports increase API surface area without clear user value

5. **why minimize API surface area?**
   → because larger APIs have more maintenance burden and more ways to misuse

**root reason**: minimal public API is easier to maintain and harder to misuse

---

## decision.6: use treestruct format for guidance messages

### 5 whys

1. **why use treestruct format with │ ├─ └─ characters?**
   → because it provides clear visual hierarchy for multi-step processes

2. **why does visual hierarchy matter?**
   → because transfer has 10 steps across multiple systems — easy to get lost

3. **why not use markdown bullets?**
   → because markdown may not render in all terminals where declastruct runs

4. **why does terminal render matter?**
   → because declastruct is a CLI tool, errors appear in terminal

5. **why is this the established pattern?**
   → because other helpful-errors in the ecosystem use treestruct for consistency

**root reason**: treestruct renders reliably in terminals and matches ecosystem patterns

---

## decision.7: integration test only for getOneDomainWhoisRecord

### 5 whys

1. **why no unit test for getOneDomainWhoisRecord?**
   → because unit test would require mock of cloudflare client

2. **why are mocks problematic?**
   → because mocks hide real API behavior differences

3. **why does real API behavior matter?**
   → because WHOIS response structure may vary by TLD or registrar

4. **why not use a fake instead of mock?**
   → because WHOIS has no business logic to fake — it's pure API call + transform

5. **why is the transform covered?**
   → because castIntoDeclaredCloudflareDomainWhoisRecord has its own unit test

**root reason**: integration test for API call + unit test for transform = full coverage

## decision.8: findsert/upsert pattern for registration

### 5 whys

1. **why use findsert/upsert for setDomainRegistration?**
   → because it matches the pattern used for zones and DNS records

2. **why use consistent patterns?**
   → because consistency reduces cognitive load for users

3. **why does findsert throw when not found (instead of create)?**
   → because cloudflare API cannot create registrations programmatically

4. **why can't cloudflare API create registrations?**
   → because domain registration requires payment, verification, and legal agreements

5. **why is throw BadRequestError the right response?**
   → because it guides users to the manual steps required to complete registration

**root reason**: findsert/upsert pattern is consistent, but registration has unique constraints

### code review revealed forbidden term in extant code

reviewed `src/domain.operations/domainDnsRecord/resolveZoneRef.ts`:

```typescript
// extant code (forbidden term):

  zoneRef: Ref<typeof DeclaredCloudflareDomainZone>,
  context: ContextCloudflareApi,

  // ...

    └── helper to expand zone ref to zone id
```
