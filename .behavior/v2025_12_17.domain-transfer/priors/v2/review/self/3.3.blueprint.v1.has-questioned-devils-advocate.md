# self-review: has-questioned-devils-advocate

argue against this blueprint — play devil's advocate.

---

## argument.1: WHOIS adds complexity for minimal value

**skeptic says**: "WHOIS lookup adds a new API call, new domain object, new cast function, new tests — all to show a slightly better error message. just tell users to check if domain is registered themselves."

**counter-argument**: the guidance difference between "transfer from godaddy" and "domain available to purchase" is significant. users who see generic "not found" will be confused about next steps. WHOIS detection is 4 files, ~100 lines of code, and prevents hours of user confusion.

**verdict**: complexity is justified by user experience improvement.

---

## argument.2: findsert diff check is over-built

**skeptic says**: "why throw BadRequestError when attributes differ? just let users re-apply to get the final state."

**counter-argument**: per rule.require.idempotency, findsert means "find extant or insert new". if found with different attributes, that's a conflict the user should know about. silent mismatch leads to confusion: "why doesn't my config match what I declared?"

**verdict**: diff check is required by idempotency semantics, not over-built.

---

## argument.3: BadRequestError for guidance is wrong

**skeptic says**: "BadRequestError implies the user made a mistake. but 'domain not in cloudflare' isn't a mistake — it's just a state. maybe use a different error type?"

**counter-argument**: BadRequestError.throw with guidance is the established helpful-errors pattern. the message body communicates "this is a state you can fix" not "you made a mistake". UnexpectedCodePathError would be worse — it implies a bug. BadRequestError is semantically correct: the request (set registration) cannot be fulfilled in current state.

**verdict**: BadRequestError is the right choice.

---

## argument.4: 10-step transfer guidance is too detailed

**skeptic says**: "that's a lot of steps to read in an error message. just link to cloudflare docs."

**counter-argument**: cloudflare docs require context-switch. step-by-step in the error message is actionable without the user to leave terminal. users can follow along. the treestruct format makes it scannable. detailed guidance reduces support burden.

**verdict**: detailed guidance is a feature, not a bug.

---

## argument.5: registrar-specific guides are maintenance burden

**skeptic says**: "vision mentions godaddy/squarespace/route53 specific guides. these will get outdated. just show generic steps."

**counter-argument**: blueprint uses generic steps with registrar name from WHOIS. no registrar-specific code paths. the WHOIS `registrarName` field personalizes the message without conditional logic. "at godaddy (current registrar)" is more helpful than "at current registrar" with zero maintenance cost.

**verdict**: registrar name from WHOIS is not a maintenance burden.

---

## argument.6: re-review all extant files is overkill

**skeptic says**: "you found a few issues in extant code. that doesn't mean every file has problems. just fix the known issues."

**counter-argument**: comprehensive skeptical code review found **19 distinct defects across 14 files**:

| category | count | examples |
|----------|-------|----------|
| findsert no diff check | 3 | setDomainZone, setDomainDnsRecord, setDomainRegistration |
| forbidden term fn/import | 6 | resolveZoneRef.ts, imports in set/get/del/getAll operations |
| forbidden term in error msg | 2 | delDomainZone, delDomainDnsRecord |
| wrong error type | 1 | setDomainRegistration throws UnexpectedCodePathError |
| forbidden `as` casts | 5 | getOneDomainZone, delDomainZone, getOneDomainRegistration, castInto* fns, DnsRecordDao |
| Ref instead of RefByUnique | 2 | DeclaredCloudflareDomainDnsRecord, DeclaredCloudflareDomainRegistration |

this demonstrates the pattern: prior implementation didn't follow current standards throughout. the cost of re-review is low (read + verify). the cost of shipped defects is high.

see `has-questioned-5whys.md` appendix for full defect list with line numbers.

**verdict**: re-review is necessary, not overkill — 19 defects across 14 files prove it.

---

## argument.7: why not automate transfers via puppeteer/playwright?

**skeptic says**: "cloudflare dashboard is just a web UI. scrape it with puppeteer and automate the transfer."

**counter-argument**:
1. scrape-based automation of cloudflare dashboard violates ToS
2. UI changes would break automation frequently
3. puppeteer adds heavy dependency
4. security risk: stored credentials for browser automation
5. cloudflare explicitly designed registrar actions to require manual verification

**verdict**: guidance is the right approach for non-automatable operations.

---

## argument.8: why DomainLiteral for WhoisRecord instead of plain object?

**skeptic says**: "WhoisRecord is just 4 fields. why create a domain object? plain object is simpler."

**counter-argument**: DomainLiteral provides:
1. construction validation via schema
2. type identity (`instanceof` checks work)
3. consistent pattern with other Declared* objects
4. enables unit test for construction
5. self-documented shape

the overhead is one file + one test file. consistent pattern wins over minimal simplicity.

**verdict**: DomainLiteral is worth the small overhead.

---

## blind spots exposed

### blind spot.1: we want this to work so we assume WHOIS is reliable

**reality check**: WHOIS data can be stale, privacy-protected, or malformed. we handle null registrar in guidance logic, but other edge cases may exist.

**action**: integration test should verify WHOIS response for various domain states.

### blind spot.2: we assume cloudflare API doesn't change

**reality check**: API versions change. fields get renamed. endpoints get deprecated.

**action**: integration tests pin current behavior. cast function should handle absent fields gracefully.

### blind spot.3: we assume users read error messages

**reality check**: some users will ignore the guidance and file support tickets.

**action**: beyond scope — guidance is best effort. users who ignore messages will learn.
