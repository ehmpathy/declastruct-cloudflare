# premortem review: 1.vision

## the scenario

it is 6 months from now. declastruct-cloudflare was released. it failed miserably. adoption is near zero. the team is discouraged. what went wrong?

---

## why did it fail?

### failure mode 1: cloudflare api did not support what we assumed

**what happened**: we built the entire zone/dns layer only to discover cloudflare's api has quirks we did not anticipate:
- some dns record types require different endpoints
- rate limits blocked bulk operations
- the `/intel/whois` endpoint required a premium plan ($200/month minimum)

**why we missed it**: we wrote the vision without deep api research. the "needs verification" items in the devil's advocate review were never verified before we started to build.

**lesson**: verify api capabilities in research phase, not implementation phase.

### failure mode 2: no one understood what "guided" meant

**what happened**: users expected full automation. they read "manage domain transfers" and assumed the tool would handle transfers. when they got a HelpfulError with 10 manual steps, they felt deceived.

negative reviews on npm: "bait and switch" "just use terraform" "does not actually automate transfers"

**why we missed it**: we knew the limitation but framed it positively. promotional language crept in.

**lesson**: be blunt about limitations upfront. "this tool cannot automate domain transfers. here's why."

### failure mode 3: terraform was good enough

**what happened**: teams evaluated declastruct-cloudflare vs terraform. terraform had:
- years of community modules
- state management solved
- docs and tutorials everywhere
- already in their ci/cd

declastruct-cloudflare had:
- typescript types (nice but not essential)
- prettier errors (nice but not essential)

teams chose terraform.

**why we missed it**: we assumed typescript-native was a killer feature. for most teams, it was a nice-to-have.

**lesson**: identify the wedge — the unique capability terraform cannot match. if that wedge does not exist, reconsider.

### failure mode 4: state management was never solved

**what happened**: users ran `declastruct apply`, made manual changes in dashboard, then ran `declastruct apply` again. the tool had no way to know what it previously created vs what was manual. chaos ensued:
- duplicate records
- deleted manual records
- support tickets flooded in

**why we missed it**: the vision hand-waved "drift detection" without a design for state. terraform solved this with `.tfstate` but we assumed it would "just work."

**lesson**: state management is hard. design it explicitly or adopt terraform's approach.

### failure mode 5: scope mismatch destroyed credibility

**what happened**: users saw "declastruct-cloudflare" and expected workers, pages, r2, d1. they got domains only. initial excitement turned to "oh, it's just dns" disappointment.

**why we missed it**: we named the repo broadly because we planned to expand. users don't care about plans.

**lesson**: name reflects current scope, not future scope. `declastruct-cloudflare-domains` is honest.

---

## what assumption was wrong?

| assumption | reality |
|------------|---------|
| typescript-native is a killer feature | nice-to-have for most teams |
| guided errors provide enough value | users felt misled about automation |
| we can compete with terraform | terraform's moat is deep |
| state management is optional | without state, drift detection is incomplete |
| broad scope name is fine | users felt baited |

---

## what risk did we ignore?

1. **api capability risk** — assumed cloudflare api supports our semantics
2. **market position risk** — assumed a niche exists between terraform and dashboard
3. **expectation risk** — assumed "guided" would be seen as a feature, not a limitation
4. **maintenance risk** — cloudflare api changes could break us; we had no relationship with cloudflare team
5. **adoption risk** — assumed developers would switch tools for marginal benefits

---

## how do we prevent these failures?

### before we proceed to blueprint

1. **verify api** — do a spike: create zone, create dns record, delete both. confirm findsert/upsert semantics work. check whois endpoint access. document quirks.

2. **be honest about scope** — rename to `declastruct-cloudflare-domains` or similar. set expectations correctly.

3. **be honest about limitations** — in readme, state clearly: "domain registration and transfer cannot be automated. this tool provides guidance only."

4. **design state** — before implementation, decide: do we track state? if so, how? if not, how do we detect drift?

5. **find the wedge** — articulate clearly: why use this over terraform? if answer is "nicer dx in typescript" — is that enough? maybe the wedge is the HelpfulError pattern for guided operations. lean into that.

## articulation: does the vision hold?

### what i found

five failure modes that could destroy this project:

1. **api assumptions unverified** — we claim findsert/upsert semantics but have not confirmed cloudflare's api supports them
2. **"guided" is misleading** — vision implies automation but registration/transfer require manual steps
3. **terraform competition unaddressed** — vision does not articulate why someone would choose this over terraform
4. **state management absent** — vision claims drift detection but no state design exists
5. **name scope mismatch** — "declastruct-cloudflare" sets wrong expectations

### do these block the vision?

| failure mode | blocks vision? | resolution |
|--------------|----------------|------------|
| api unverified | NO | research phase verifies before blueprint |
| "guided" misleading | YES | vision should be more explicit |
| terraform competition | NO | blueprint phase addresses differentiation |
| state management | NO | blueprint phase designs state strategy |
| name mismatch | NO | can rename anytime before release |

### what must change in vision

**the vision needs one fix**: be more explicit that registration/transfer are **not automated**.

the current vision says:
> "register new domain | `setDomainRegistration({ findsert })` | guided (HelpfulError)"

this is honest but buried. the day-in-the-life example should make this clearer upfront — not as a surprise at apply time.

### why the vision holds despite these issues

the vision holds because:

1. **it paints the right picture** — even with "guided" limitations, the dx improvement over dashboard-only is real
2. **it identifies the right domain objects** — zone, dns record, registration separation is correct
3. **it sets up research questions** — the "needs verification" items are captured for next phase
4. **it acknowledges awkwardness** — the "what's awkward" section is honest

the issues i found are **design questions for later phases**, not **vision flaws**. the one exception — "guided" clarity — is a documentation improvement, not a directional change.

### verdict

**the vision holds.** proceed to research phase with these items on the radar:
- verify api capabilities (research phase)
- design state strategy (blueprint phase)
- articulate terraform differentiation (blueprint phase)
- consider rename to scope (pre-release decision)
