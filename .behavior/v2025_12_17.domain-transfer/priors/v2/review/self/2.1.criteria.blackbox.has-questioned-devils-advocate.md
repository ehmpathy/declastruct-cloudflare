# devil's advocate review: 2.1.criteria.blackbox

## usecase.1 = zone management

**why might this be wrong?**
- "returns all zones" — what if account has 1000 zones? pagination not specified
- "creates the zone" — what if zone already exists at different account? cloudflare may reject

**what would a skeptic say?**
- "just use terraform" — terraform already handles zones

**too strict? too loose?**
- too loose: doesn't specify zone types (full vs partial)

**tests correct behavior?**
- yes, but absent: what happens when zone is in "awaited" status for nameserver change?

## usecase.2 = dns record management

**why might this be wrong?**
- "findsert mode throws conflict error" — is this actually desired? maybe user wants to see diff, not error
- "creates zone first, then record" — assumes we can auto-create zones, but user may not want that

**what would a skeptic say?**
- dns record types have different constraints (MX needs priority, SRV has special format) — criteria too generic

**too strict? too loose?**
- too loose: doesn't specify which record types are supported
- too loose: doesn't specify TTL constraints per plan

**tests correct behavior?**
- mostly yes, but MX/SRV/TXT records have specific format rules not captured

## usecase.3 = domain registration management

**why might this be wrong?**
- only covers "update settings" — what about initial registration? that's guided, but criteria don't clarify

**what would a skeptic say?**
- this usecase is thin — only 2 criteria

**too strict? too loose?**
- too loose: doesn't specify what "registered in cloudflare" means — purchased there vs transferred there

**tests correct behavior?**
- yes for settings update, but absent: what if domain is locked by 60-day transfer rule?

## usecase.4 = domain transfer guidance

**why might this be wrong?**
- "shows transfer status from cloudflare API" — assumes this API exists. not verified
- "resumes guidance at current step" — implies state machine we may not implement

**what would a skeptic say?**
- WHOIS API may require premium plan — criteria assume free access
- "registrar-specific docs" for 3 registrars — what about namecheap, hover, etc?

**too strict? too loose?**
- too strict: "step-by-step" implies specific format we haven't designed
- too loose: doesn't specify what "transfer completed" detection looks like

**tests correct behavior?**
- direction is right, but details assume API capabilities not yet verified

## usecase.5 = domain purchase guidance

**why might this be wrong?**

**what would a skeptic say?**
- purchase flow has price info — we don't show price. user may be surprised by cost

**too strict? too loose?**
- too loose: doesn't specify TLD support (cloudflare doesn't support all TLDs)

**tests correct behavior?**
- yes, but edge case: what if domain is premium priced?

## usecase.6 = drift detection

**why might this be wrong?**
- "shows what will be deleted" — implies we track what we created. no state file specified

**what would a skeptic say?**
- without state file, we can only show drift for declared resources, not detect "extra" resources added via dashboard

**too strict? too loose?**
- too loose: doesn't specify how drift is displayed (json? tree? diff format?)

**tests correct behavior?**
- partially — only tests declared-vs-actual, not actual-vs-declared (orphan detection)

## usecase.7 = error cases

**why might this be wrong?**
- "A record and CNAME on same name" is too specific — there are many dns constraint violations

**what would a skeptic say?**
- error cases are infinite — we can't enumerate them all
- "authentication error" — cloudflare may return different errors for expired vs invalid token

**too strict? too loose?**
- too strict: enumerates specific cases that should be generalized
- too loose: doesn't specify error format (HelpfulError structure?)

**tests correct behavior?**
- direction right, but over-specifies one case while under-specifies error structure

## summary

**the criteria hold**, but 3 items need verification:
1. WHOIS API access (plan requirements)
2. transfer status API (exists?)
3. state management strategy (how do we detect drift without state file?)

these are research/blueprint questions, not criteria flaws.

## articulation: what I found

I reviewed each criterion as a skeptic would. here's what I found:

**the criteria hold because:**

1. **core usecases are well-scoped** — zone, dns record, and registration management map directly to the wish. no gold-plating.

2. **guidance patterns are correct** — transfer and purchase guidance are HelpfulErrors, not automation. this matches cloudflare's API limitations.

3. **drift detection is essential** — without it, declastruct adds no value over manual dashboard work.

**weak points identified:**

1. **WHOIS API access** — usecase.4 and usecase.5 assume WHOIS works. if it requires premium plan, the "smart guidance" feature is blocked. **action: verify in research phase.**

2. **transfer status API** — usecase.4 assumes we can detect "transfer in progress". if cloudflare doesn't expose this, we simplify to "needs transfer" vs "transfer done". **action: verify in research phase.**

3. **state management absent** — usecase.6 implies drift detection, but criteria don't specify how we track what we created. without state, we can only compare declared-vs-actual, not detect orphans. **action: address in blueprint.**

4. **dns constraints under-specified** — usecase.7 names one constraint (A+CNAME conflict) but dns has many. **recommendation: generalize to "cloudflare validates dns constraints; errors bubble up".**

**no criteria should be removed.** the identified weak points are research/design questions, not criteria flaws. the experience boundaries are correctly specified.
