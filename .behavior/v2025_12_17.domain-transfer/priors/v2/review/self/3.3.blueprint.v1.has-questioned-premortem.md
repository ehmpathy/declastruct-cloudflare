# self-review: has-questioned-premortem

imagine this blueprint was implemented and failed miserably.

---

## failure.1: WHOIS API rate limits

**what happened**: declastruct apply ran for 50 domains, hit cloudflare WHOIS API rate limits, and failed midway through.

**why it failed**: no rate limit handle in getOneDomainWhoisRecord.

**what assumption was wrong**: assumed WHOIS API has unlimited capacity.

**mitigation**: add retry with exponential backoff to getOneDomainWhoisRecord. check cloudflare docs for rate limits.

**action**: add to blueprint codepath treestruct: retry logic for WHOIS operation.

---

## failure.2: WHOIS returns null registrar for privacy-protected domains

**what happened**: user tried to transfer domain from registrar with strong WHOIS privacy. WHOIS returned `registrar: null`. guidance showed "purchase required" when it should show "transfer required".

**why it failed**: assumed WHOIS always returns registrar for registered domains.

**what assumption was wrong**: WHOIS privacy can hide registrar identity.

**mitigation**: check `found: true` + `registrar: null` case separately. show: "domain appears registered but registrar is hidden. check domain ownership via dashboard."

**action**: blueprint guidance logic already handles this in the else branch — shows purchase guidance. update to include registrar-hidden case with distinct message.

---

## failure.3: zone created but registration fails

**what happened**: declastruct created the zone successfully. then setDomainRegistration threw BadRequestError with transfer guidance. user followed steps, transferred domain, but zone was left in "wait-for-nameserver" state.

**why it failed**: zone creation and registration guidance are separate operations. no link between them.

**what assumption was wrong**: assumed users would understand they need to update nameservers at current registrar before transfer completes.

**mitigation**: include nameserver update step in transfer guidance. show cloudflare nameservers from zone.nameServers field.

**action**: update blueprint guidance message to include nameserver step before step 8 (approve FOA email).

---

## failure.4: findsert returns extant with different attributes

**what happened**: user declared zone with `type: 'full'`, but extant zone has `type: 'partial'`. findsert returned the extant zone without error. user expected it to match their declaration.

**why it failed**: findsert just returns found object without attribute diff check.

**what assumption was wrong**: assumed findsert semantics were implemented correctly in extant code.

**what integration broke**: zone findsert returns mismatched attributes silently.

**action**: this is already captured in has-questioned-assumptions. re-review must add diff check.

---

## failure.5: dns record created before zone activation

**what happened**: user declared zone + dns records. declastruct created zone (status: "queued"). then created dns records. records were accepted but not propagated because zone not active yet.

**why it failed**: declastruct does not wait for zone activation.

**what assumption was wrong**: assumed dns records work immediately after zone creation.

**mitigation**: document that dns records require zone activation. or: setDomainDnsRecord checks zone status and throws guidance if not active.

**action**: add note to blueprint about zone activation dependency. consider whether setDomainDnsRecord should check zone status.

---

## failure.6: registration update with incompatible settings

**what happened**: user declared `locked: false` on a domain within 60 days of transfer. cloudflare API rejected it because ICANN requires 60-day lock.

**why it failed**: no validation of ICANN constraints.

**what assumption was wrong**: assumed all registration settings can be freely changed.

**mitigation**: setDomainRegistration should check transfer date and warn about 60-day lock constraint.

**action**: add to blueprint: validation for ICANN constraints in setDomainRegistration upsert path.

---

## failure.7: cast function fails on unexpected API response

**what happened**: cloudflare updated their WHOIS API response structure. castIntoDeclaredCloudflareDomainWhoisRecord threw because expected fields were absent.

**why it failed**: cast function assumes specific response shape.

**what assumption was wrong**: assumed API response shape is stable.

**mitigation**: cast function should handle absent fields gracefully. return `found: false` if response is malformed.

**action**: add defensive handle to cast function. integration test verifies current API response shape.

---

## failure.8: test suite passes but production fails

**what happened**: all tests passed in CI. deployed to production. WHOIS API returned different data for production TLDs than test TLDs.

**why it failed**: integration tests only cover test domains.

**what assumption was wrong**: assumed test domain WHOIS behavior matches production domains.

**mitigation**: integration tests should cover multiple TLDs (.com, .net, .org, .io, ccTLDs).

**action**: add to blueprint test coverage: integration tests for various TLDs.

## what scale issue emerged?

at scale (100+ domains), WHOIS rate limits become critical. sequential WHOIS calls for each domain would be slow and hit limits.

**mitigation**: document that large-scale domain management should batch operations or use cloudflare dashboard for bulk transfers.
