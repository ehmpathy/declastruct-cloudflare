# requirements review: 2.1.criteria.blackbox

## usecase.1 = zone management

**who said this was needed?** the wish explicitly asks for "get/set operations to see what domains are already in cloudflare"

**what if removed?** users couldn't manage zones via code — core to the vision

**scope right?** yes — covers list, create, delete with idempotency

**simpler test?** no — these are atomic operations already

**verdict**: required

---

## usecase.2 = dns record management

**who said this was needed?** the wish asks for "get/set operations to manage the dns of those domains"

**what if removed?** users couldn't manage dns records — the primary use case

**scope right?** yes — covers CRUD with findsert/upsert semantics

**simpler test?** the findsert vs upsert distinction may be too detailed for blackbox — but it's essential to the declastruct pattern

**verdict**: required

---

## usecase.3 = domain registration management

**who said this was needed?** the wish asks for operations to "register new ones"

**what if removed?** users couldn't manage registration settings like autoRenew

**scope right?** yes — minimal, focused on settings

**simpler test?** already minimal

**verdict**: required

---

## usecase.4 = domain transfer guidance

**who said this was needed?** the wish explicitly asks for "get/set operations to manage the import of domains into cloudflare, transferred out of another registrar"

**what if removed?** users would get cryptic errors when domains need transfer — core differentiator

**scope right?** the "in progress" and "completed" states may be premature — we haven't verified cloudflare API exposes transfer status

**question**: do we need to track transfer progress, or just detect "needs transfer" vs "already transferred"?

**verdict**: required, but "in progress" status may be out of scope for v1

---

## usecase.5 = domain purchase guidance

**who said this was needed?** implied by "register new ones" in wish

**what if removed?** users would get unclear errors when domain is unregistered

**scope right?** yes — parallel to transfer guidance

**simpler test?** already minimal

**verdict**: required

---

## usecase.6 = drift detection

**who said this was needed?** the vision mentions "declastruct plan shows drift" as key benefit

**what if removed?** users couldn't preview changes before apply — standard declastruct pattern

**scope right?** yes

**simpler test?** already minimal

**verdict**: required — core to declastruct pattern

---

## usecase.7 = error cases

**who said this was needed?** implicit in any robust system

**what if removed?** users would get unhelpful errors

**scope right?** maybe too specific — "A record and CNAME on same name" is one of many dns constraints

**question**: should we enumerate all dns constraint errors, or just say "dns constraint violations bubble up"?

**verdict**: required, but dns constraint detail may be over-specified

---

## summary

| usecase | verdict | notes |
|---------|---------|-------|
| zone management | required | core wish |
| dns record management | required | core wish |
| registration management | required | core wish |
| transfer guidance | required | may simplify "in progress" state |
| purchase guidance | required | parallel to transfer |
| drift detection | required | declastruct pattern |
| error cases | required | may generalize dns constraints |

**no criteria should be removed.** two may be simplified:
1. usecase.4: "in progress" state depends on API capability — verify in research
2. usecase.7: dns constraint errors could be generalized to "cloudflare rejects invalid combinations"
