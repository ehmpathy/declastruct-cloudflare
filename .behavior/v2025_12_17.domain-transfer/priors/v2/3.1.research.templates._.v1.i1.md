# research.templates: declastruct provider patterns

## overview

this document analyzes templates from other declastruct providers to extract key patterns for declastruct-cloudflare implementation.

---

## template.1: declastruct-github

**repo**: https://github.com/ehmpathy/declastruct-github [1]

**description** [1]:
> "declarative control of GitHub constructs via declastruct - manage repos, branches, and protection rules"

### src structure

```
src/
├── .test/
├── access/
├── contract/
├── domain.objects/
├── domain.operations/
└── domain/
```

### key pattern: wish file structure

**quote** [1]:
```typescript
export const getProviders = async (): Promise<DeclastructProvider[]> => [
  getDeclastructGithubProvider(
    {
      credentials: {
        token: process.env.GITHUB_TOKEN ??
          UnexpectedCodePathError.throw('github token not supplied'),
      },
    },
    { log: console },
  ),
];

export const getResources = async (): Promise<DomainEntity<any>[]> => {
  const repo = DeclaredGithubRepo.as({
    owner: 'ehmpathy',
    name: 'declastruct-github',
    description: '...',
    visibility: 'public',
  });
  return [repo, repoConfig];
};
```

**relation to wish**: same pattern needed for cloudflare — `getProviders()` + `getResources()` exports

### key pattern: domain entity composition

**quote** [1]:
```typescript
// declare the repo
const repo = DeclaredGithubRepo.as({
  owner: 'ehmpathy',
  name: 'declastruct-github',
});

// declare config for the repo
const repoConfig = DeclaredGithubRepoConfig.as({
  repo,  // reference to repo instance
  defaultBranch: 'main',
  hasIssues: true,
  // ...
});
```

**relation to wish**: similar composition needed — `DeclaredCloudflareDomainDnsRecord` references `DeclaredCloudflareDomainZone`

### key pattern: cli workflow

**quote** [1]:
```sh
# plan
npx declastruct plan --wish provision/github/resources.ts --output plan.json

# apply
npx declastruct apply --plan plan.json
```

**relation to wish**: identical workflow for cloudflare domain management


**relation to wish**: similar ref pattern for dns record → zone reference

## cross-template pattern summary

### pattern.1: provider factory

**all templates** export a `getDeclastructXxxProvider()` function:

```typescript
export const getDeclastructCloudflareProvider = async (
  input: { apiToken: string; accountId: string },
  context: ContextLogTrail,
): Promise<DeclastructProvider> => {
  const client = new Cloudflare({ apiToken: input.apiToken });
  const daos = { DeclaredCloudflareDomainZone: ..., ... };
  return new DeclastructProvider({ name: 'cloudflare', daos, context });
};
```

### pattern.2: domain objects with static keys

**all templates** define domain objects with `static primary`, `static unique`:

```typescript
class DeclaredXxx extends DomainEntity<DeclaredXxx> {
  public static primary = ['id'] as const;
  public static unique = ['name'] as const;
  public static metadata = ['id'] as const;
}
```

### pattern.3: set operations with PickOne

**all templates** use `PickOne<{ findsert; upsert }>` for set operations:

```typescript
const setXxx = async (
  input: PickOne<{ findsert: Xxx; upsert: Xxx }>,
  context: Context,
) => { ... };
```

### pattern.4: dao interface

**all templates** expose daos with:
- `getOne({ by: { primary | unique | ref } })`
- `getAll()`
- `set({ findsert | upsert })`
- `del({ by })`

### pattern.5: reference composition

**all templates** compose entities via refs:
- github: `repoConfig.repo = repo`
- stripe: `invoiceItem.invoiceRef = { id: invoice.id }`
- cloudflare: `dnsRecord.zone = { name: zone.name }`
