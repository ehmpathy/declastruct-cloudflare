the fact that we can set it via dashboard means its NOT readonly

readonly means that it cant be set anywhere by anyone other than the database

if the caller has control over it, its not readonly, its writable (we just have to offer guidance on how to write it)

its critical to ensure that all writeable fields are NOT marked readonly, because this will allow declastruct to inform the user if something is out of sync and they need to go in and manually make a change in the dashboard (and enable us to trigger the guidance of how to do that, in the set operation, via a HelpfulError)

plus, that way when an api operation does eventually come out, we can just drop the error and do it for them
