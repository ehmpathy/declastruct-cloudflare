# blackbox criteria: declastruct-cloudflare domain management

blackbox = experience boundaries (no implementation details)

---

# usecase.1 = provider setup

given a user with cloudflare api credentials
  when they initialize the declastruct cloudflare provider
    then they receive a provider instance
      sothat they can use it with declastruct plan/apply

given a user with invalid cloudflare api credentials
  when they initialize the declastruct cloudflare provider
    then they receive a clear authentication error
      sothat they know to fix their credentials

---

# usecase.2 = zone management

## episode: declare and apply zones

given a user who wants to add a domain to cloudflare
  when they declare a zone in their resources file
    then the zone declaration is accepted
      sothat they can proceed to plan/apply

given a user with a declared zone
  when they run declastruct plan
    then they see the zone will be created
      sothat they can review before apply

given a user who runs declastruct apply with a new zone
  when the apply completes
    then the zone exists in cloudflare
      sothat dns records can be added to it

## exchange: get zones

given zones exist in the cloudflare account
  when getAllDomainZones is called
    then all zones are returned
      sothat users can see what domains are in cloudflare

given no zones exist in the cloudflare account
  when getAllDomainZones is called
    then an empty list is returned
      sothat users know the account has no zones

## exchange: set zone

given a zone does not exist
  when setDomainZone is called with findsert
    then the zone is created
      sothat the domain is added to cloudflare

given a zone already exists
  when setDomainZone is called with findsert
    then the zone is returned without modification
      sothat the operation is idempotent

given a zone already exists
  when setDomainZone is called with upsert
    then the zone is updated with new settings
      sothat configuration changes are applied

## exchange: delete zone

given a zone exists
  when delDomainZone is called
    then the zone is removed from cloudflare
      sothat the domain is no longer managed

given a zone does not exist
  when delDomainZone is called
    then no error is raised
      sothat the operation is idempotent

# usecase.3 = dns record management

## episode: declare and apply dns records

given a user who wants to add dns records
  when they declare records in their resources file
    then the record declarations are accepted
      sothat they can proceed to plan/apply

given a user with declared dns records that reference a zone
  when they run declastruct plan
    then they see which records will be created
      sothat they can review before apply

given a user who runs declastruct apply with dns records
  when the apply completes
    then the records exist in cloudflare
      sothat the domain resolves correctly

## exchange: get dns records

given dns records exist for a zone
  when getAllDomainDnsRecords is called with that zone
    then all records for the zone are returned
      sothat users can see what records exist

given no dns records exist for a zone
  when getAllDomainDnsRecords is called with that zone
    then an empty list is returned
      sothat users know the zone has no records

## exchange: set dns record

given a dns record does not exist
  when setDomainDnsRecord is called with findsert
    then the record is created
      sothat the dns entry is added

given a dns record already exists with the same unique key
  when setDomainDnsRecord is called with findsert
    then the record is returned without modification
      sothat the operation is idempotent

given a dns record already exists
  when setDomainDnsRecord is called with upsert
    then the record is updated with new values
      sothat configuration changes are applied

given multiple A records for load distribution are declared
  when setDomainDnsRecord is called for each
    then all records are created
      sothat round-robin load distribution works

## exchange: delete dns record

given a dns record exists
  when delDomainDnsRecord is called
    then the record is removed from cloudflare
      sothat the dns entry no longer resolves

given a dns record does not exist
  when delDomainDnsRecord is called
    then no error is raised
      sothat the operation is idempotent

## edge case: record type conflicts

given an A record exists for a name
  when a CNAME is declared for the same name
    then cloudflare rejects the record due to conflict
      sothat users are prevented from invalid dns state

# usecase.5 = domain transfer guidance

## episode: transfer domain from another registrar

given a domain is registered elsewhere (e.g., godaddy)
  when setDomainRegistration is called for that domain
    then a BadRequestError is raised with transfer guidance
      sothat the user knows exactly how to transfer

given the transfer guidance is displayed
  when the user reads the error message
    then they see step-by-step instructions for both registrars
      sothat they can complete the transfer manually

given the transfer guidance includes registrar-specific steps
  when the user follows the steps at the current registrar
    then they can unlock the domain and get the auth code
      sothat the transfer can proceed

given the user has completed the manual transfer steps
  when they re-run declastruct apply after transfer completes
    then setDomainRegistration succeeds
      sothat registration settings are applied

## edge case: domain is unregistered

given a domain is not registered anywhere
  when setDomainRegistration is called for that domain
    then a BadRequestError is raised with purchase guidance
      sothat the user knows to register it via cloudflare dashboard

given the purchase guidance is displayed
  when the user reads the error message
    then they see steps to register via cloudflare dashboard
      sothat they can purchase the domain manually

# usecase.6 = the plan/apply workflow

## episode: declare -> plan -> apply

given a user has written a resources.ts file
  when they run declastruct plan
    then a plan.json is generated that shows all changes
      sothat they can review before apply

given a user reviews the plan
  when they run declastruct apply
    then the changes are applied in the correct order
      sothat zones are created before records that reference them

given a user runs declastruct apply twice

    then no changes are made
      sothat the system is idempotent

## episode: drift detection

  when the user runs declastruct plan
    then the plan shows the drift
      sothat they can reconcile code with reality

# boundary conditions

## zone boundaries

- zone name must be a valid domain (e.g., "example.com")
- zone type must be "full" or "partial"

## dns record boundaries

- record name must be within the zone (e.g., "api.example.com" for zone "example.com")
- record type must be valid (A, AAAA, CNAME, MX, TXT, etc.)
- ttl must be within cloudflare limits (minimum 60s for most plans)
- proxied is only valid for A, AAAA, and CNAME records

## registration boundaries

- domain must be a registrable tld supported by cloudflare
- autoRenew, locked, privacyProtection are boolean settings
- some settings may not apply to all tlds
