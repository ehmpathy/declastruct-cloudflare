research on open source tools to leverage for declastruct-cloudflare production codepaths.

## objective

identify maintained, battle-tested OSS packages that solve parts of our problem space.

## lever.1: cloudflare (official sdk)

### citations

[1] https://www.npmjs.com/package/cloudflare

> "The official TypeScript library for the Cloudflare API"

[2] https://github.com/cloudflare/cloudflare-typescript

> "This library is generated from our OpenAPI spec"

> "Node.js 18 LTS or later (non-EOL) versions are supported"

### assessment

| attribute | value |
|-----------|-------|
| maintainer | cloudflare (official) |
| weekly downloads | ~200k |
| last publish | actively maintained |
| typescript | native (generated from openapi) |
| node version | 18+ |

### pros

- official sdk — guaranteed api parity
- typescript native — full type safety
- auto-generated — stays current with api changes
- well documented — mirrors api docs

### cons

- large surface area — includes all cloudflare apis (not just domains)
- generated code style — may differ from hand-crafted patterns

### usage example

```ts
import Cloudflare from 'cloudflare';

const client = new Cloudflare({
  apiToken: process.env.CLOUDFLARE_API_TOKEN,
});

// list zones
const zones = await client.zones.list({
  account: { id: accountId },
});

// create dns record
const record = await client.dns.records.create({
  zone_id: zoneId,
  type: 'A',
  name: 'api.example.com',
  content: '192.168.1.100',
  ttl: 300,
  proxied: true,
});
```

### verdict

**must use** — official sdk is the foundation for all cloudflare api interactions.

## lever.2: domain-objects

### citations

[3] https://www.npmjs.com/package/domain-objects

> "Represent domain objects, leverage domain knowledge, and add validation"

[4] https://github.com/ehmpathy/domain-objects

> "DomainLiteral: immutable, identity defined by all properties"

> "DomainEntity: mutable, identity defined by unique properties"

### assessment

| attribute | value |
|-----------|-------|
| maintainer | ehmpathy |
| weekly downloads | ~5k |
| last publish | actively maintained |
| typescript | native |
| used by | declastruct ecosystem |

### pros

- DDD patterns — DomainEntity, DomainLiteral, DomainEvent
- reference system — RefByUnique, RefByPrimary, refByUnique()
- unique/primary keys — static declarations for identity
- serialization — getUniqueIdentifier, serialize
- nested hydration — automatic instantiation of nested objects

### cons

- learning curve — requires understanding DDD concepts
- ehmpathy ecosystem — not as widely adopted as generic alternatives

### usage example

```ts
import { DomainEntity, RefByUnique, refByUnique } from 'domain-objects';

interface DeclaredCloudflareDomainZone {
  id?: string;
  name: string;
  type: 'full' | 'partial';
}

class DeclaredCloudflareDomainZone
  extends DomainEntity<DeclaredCloudflareDomainZone>
  implements DeclaredCloudflareDomainZone
{
  public static primary = ['id'] as const;
  public static unique = ['name'] as const;
}

// create instance
const zone = new DeclaredCloudflareDomainZone({
  name: 'example.com',
  type: 'full',
});

// extract reference
const zoneRef = refByUnique(zone);
// zoneRef = { name: 'example.com' }
```

### verdict

**must use** — core of declastruct pattern, provides domain object infrastructure.

## lever.3: helpful-errors

### citations

[5] https://www.npmjs.com/package/helpful-errors

> "A collection of helpful errors"

[6] https://github.com/ehmpathy/helpful-errors

> "BadRequestError, UnexpectedCodePathError, NotFoundError, NotAuthorizedError"

### assessment

| attribute | value |
|-----------|-------|
| maintainer | ehmpathy |
| weekly downloads | ~3k |
| last publish | actively maintained |
| typescript | native |
| used by | declastruct ecosystem |

### pros

- structured errors — metadata, cause chain, helpful messages
- semantic types — BadRequestError, UnexpectedCodePathError
- static throw — `BadRequestError.throw(message, metadata)`
- wrap pattern — `HelpfulError.wrap(fn, context)()`
- serialization — toJSON for logs

### cons

- ehmpathy ecosystem — not as widely adopted

### usage example

```ts
import { BadRequestError, UnexpectedCodePathError } from 'helpful-errors';

// throw with guidance
BadRequestError.throw(
  `domain "example.com" not found in cloudflare registrar

⚠️ transfer required
   └─ steps
      ├─ 1. unlock domain at current registrar
      └─ 2. initiate transfer at cloudflare`,
  { domain: 'example.com', registrar: 'godaddy' },
);

// wrap for context
await HelpfulError.wrap(
  async () => await cloudflare.zones.create({ name }),
  { message: 'failed to create zone', metadata: { name } },
)();
```

### verdict

**must use** — provides error patterns for transfer/purchase guidance.

## lever.4: type-fns

### citations

[7] https://www.npmjs.com/package/type-fns

> "A collection of type functions for TypeScript"

[8] https://github.com/ehmpathy/type-fns

> "PickOne, HasMetadata, isPresent, omit, pick"

### assessment

| attribute | value |
|-----------|-------|
| maintainer | ehmpathy |
| weekly downloads | ~5k |
| last publish | actively maintained |
| typescript | native |

### pros

- type utilities — PickOne for exclusive unions
- HasMetadata — mark objects with readonly fields populated
- runtime utils — isPresent, omit, pick
- zero dependencies

### cons

- small scope — focused utility library

### usage example

```ts
import { PickOne, HasMetadata, isPresent } from 'type-fns';

// exclusive input variants
const setDomainZone = async (
  input: PickOne<{
    findsert: DeclaredCloudflareDomainZone;
    upsert: DeclaredCloudflareDomainZone;
  }>,
) => { ... };

// mark readonly fields populated
type ZoneWithMetadata = HasMetadata<DeclaredCloudflareDomainZone>;

// filter nulls with type narrowing
const records = [record1, null, record2].filter(isPresent);
// records: DnsRecord[]
```

### verdict

**must use** — provides type utilities for findsert/upsert pattern.

## alternatives considered

### alternative: raw fetch vs cloudflare sdk

| approach | pros | cons |
|----------|------|------|
| raw fetch | minimal deps | no types, manual error handle |
| cloudflare sdk | full types, maintained | larger bundle |

**decision**: use cloudflare sdk — types and maintenance outweigh bundle size

### alternative: zod vs domain-objects validation

| approach | pros | cons |
|----------|------|------|
| zod | popular, standalone | no identity semantics |
| domain-objects | ddd patterns, refs | ecosystem specific |

**decision**: use domain-objects — declastruct pattern requires identity semantics
