# research: infra patterns for test framework

## .what

research on infrastructure patterns and techniques for test frameworks that enable:
- rapid feedback loops during development
- integration tests on boundaries (boundary components against remote dependencies)
- integration tests on composites (combined components in end-to-end procedures)
- acceptance tests that prove behavior fulfillment

## .context

wish = domain transfer & management in cloudflare via declastruct declarative control

## .findings

### 1. test framework layering

the codebase uses a three-tier test strategy:

| layer | config | purpose | access | speed |
|-------|--------|---------|--------|-------|
| unit | jest.unit.config.ts | isolated logic, transforms, pure functions | internal | fast (~ms) |
| integration | jest.integration.config.ts | end-to-end with internal contracts | internal | medium (~1-10s) |
| acceptance | jest.acceptance.config.ts | end-to-end blackbox via cli only | external only | slow (~30-60s) |

**key distinctions:**
- **integration**: tests with access to internal contracts (can import from `@src/...`)
  - **atomic boundaries**: individual operations against remote dependencies (e.g., `getAllDomainZones`, `setDomainDnsRecord`)
  - **end-to-end compositions**: combined operations in procedures (e.g., `createZoneWithDnsRecords`)
- **acceptance**: end-to-end blackbox testing, only uses public SDK exports (imports from `dist/`), verifies wish/vision/criteria fulfillment

```
test pyramid:
         /\
        /  \  acceptance (few, slow, high confidence)
       /____\
      /      \  integration (moderate, medium speed)
     /________\
    /          \  unit (many, fast, low-level confidence)
   /______________\
```

### 2. rapid feedback loop patterns

#### 2.1 changedSince optimization

```sh
# only run tests affected by changes since main branch
npm run test:unit        # uses --changedSince=main by default
npm run test:integration # uses --changedSince=main by default

# force all tests (for ci, releases, or thorough verification)
THOROUGH=true npm run test:unit
THOROUGH=true npm run test:integration
```

this pattern enables:
- fast local iteration (only changed tests run)
- thorough ci validation (all tests run)

#### 2.2 credential gating

```typescript
// jest.integration.env.ts
if (!(process.env.CLOUDFLARE_ACCOUNT_ID || process.env.CLOUDFLARE_API_TOKEN))
  throw new Error('cloudflare credentials not set...');
```

fail-fast on missing credentials prevents:
- confusing errors deep in test execution
- wasted time on tests that can't succeed

#### 2.3 fail-fast on missing preconditions

```typescript
when('[t0] getOneDomainZone is called by primary key', () => {
  then('it should return the zone', async () => {
    // fail-fast if precondition not met
    if (!zonesData.firstZone)
      throw new Error(
        'test precondition not met: no zones exist in account. ' +
        'provision a test zone or use an account with existing zones.',
      );

    const zone = await getOneDomainZone({ ... });
    expect(zone).toBeDefined();
  });
});
```

**principles:**
- **never silently skip assertions** - tests should fail explicitly when preconditions aren't met
- **explicit error messages** - clearly state what's missing and how to fix it
- **self-provisioning tests** - ideally tests provision their own resources

**anti-pattern (forbidden):**
```typescript
// NEVER do this - hides missing preconditions
if (!resource) return; // silently passes without testing anything
```

### 3. integration test patterns

integration tests verify both:
1. **atomic boundaries**: individual operations against remote dependencies
2. **end-to-end compositions**: combined operations in procedures

#### 3.1 bdd structure via test-fns

```typescript
import { given, when, then, useBeforeAll } from 'test-fns';

describe('domainZone', () => {
  given('[case1] cloudflare api context', () => {
    const zonesData = useBeforeAll(async () => {
      const allZones = await getAllDomainZones(getContext());
      const firstZone = allZones[0];

      // fail-fast if precondition not met
      if (!firstZone)
        throw new Error(
          'test precondition not met: no zones exist in account. ' +
          'provision a test zone or use an account with existing zones.',
        );

      return { zones: allZones, firstZone };
    });

    when('[t0] getOneDomainZone called by primary', () => {
      then('returns the zone', async () => {
        const zone = await getOneDomainZone({
          by: { primary: { id: zonesData.firstZone.id } },
        }, getContext());
        expect(zone?.id).toEqual(zonesData.firstZone.id);
      });
    });

    when('[t1] getOneDomainZone called by unique', () => {
      then('returns the zone', async () => {
        const zone = await getOneDomainZone({
          by: { unique: { name: zonesData.firstZone.name } },
        }, getContext());
        expect(zone?.name).toEqual(zonesData.firstZone.name);
      });
    });
  });
});
```

benefits:
- readable test output: `given: cloudflare api context > when: getOneDomainZone called > then: returns the zone`
- shared setup via `useBeforeAll` (runs once per given block)
- clear case/event labeling: `[case1]`, `[t0]`, `[t1]`
- fail-fast in setup if preconditions not met (no silent skips)

#### 3.2 context factory pattern

```typescript
// src/.test/getSampleCloudflareApiContext.ts
export const getSampleCloudflareApiContext = (): ContextCloudflareApi & ContextLogTrail => {
  const apiToken = process.env.CLOUDFLARE_API_TOKEN;
  const accountId = process.env.CLOUDFLARE_ACCOUNT_ID;

  // fail-fast on missing credentials
  if (!apiToken) BadRequestError.throw('CLOUDFLARE_API_TOKEN env var required');
  if (!accountId) BadRequestError.throw('CLOUDFLARE_ACCOUNT_ID env var required');

  const client = new Cloudflare({ apiToken });
  return {
    cloudflare: { client, accountId },
    log: { debug: () => {}, info: () => {}, warn: () => {}, error: () => {} },
  };
};
```

benefits:
- dependency injection via context
- centralized credential validation
- testable (can mock context in unit tests)

#### 3.3 fail-fast credential validation

```typescript
// jest.integration.env.ts - runs before all tests
if (!(process.env.CLOUDFLARE_ACCOUNT_ID || process.env.CLOUDFLARE_API_TOKEN))
  throw new Error('cloudflare credentials not set. run: source .agent/repo=.this/role=any/skills/use.demo.cloudflare.creds.sh');
```

benefits:
- immediate failure if credentials missing
- clear error message with fix instructions
- no silent skips or confusing partial runs

**anti-pattern (forbidden):**
```typescript
// NEVER do this - hides credential issues
const hasCredentials = process.env.CLOUDFLARE_API_TOKEN && process.env.CLOUDFLARE_ACCOUNT_ID;
const describeWithCredentials = hasCredentials ? describe : describe.skip;
```

### 4. acceptance test patterns

#### 4.1 cli-based black-box testing

```typescript
describe('declastruct CLI workflow', () => {
  given('a declastruct resources file with cloudflare provider', () => {
    const testDir = join(__dirname, '.test', '.temp', 'acceptance', `run.${...}`);
    const resourcesFile = join(__dirname, '.test', 'assets', 'resources.acceptance.ts');
    const planFile = join(testDir, 'plan.json');

    when('[t0] generating a plan via declastruct CLI', () => {
      const prep = useBeforeAll(async () => {
        execSync(
          `npx declastruct plan --wish ${resourcesFile} --into ${planFile}`,
          { stdio: 'inherit', env: process.env, cwd: join(__dirname, '..', '..', '..') },
        );
        const plan = JSON.parse(readFileSync(planFile, 'utf-8'));
        return { plan };
      });

      then('creates a valid plan file', () => {
        expect(existsSync(planFile)).toBe(true);
      });

      then('plan has expected structure', () => {
        expect(prep.plan).toHaveProperty('changes');
        expect(Array.isArray(prep.plan.changes)).toBe(true);
      });
    });
  });
});
```

key patterns:
- uses `execSync` to invoke cli as a user would
- imports from `dist/` (built package, not source)
- stdio: 'inherit' for visible output
- unique temp dirs per run via timestamp

#### 4.2 idempotency verification

```typescript
when('[t2] applying the same plan twice (idempotency)', () => {
  then('second apply succeeds without error', async () => {
    // first plan
    execSync(`npx declastruct plan --wish ${resourcesFile} --into ${planFile}`, {...});

    // first apply
    execSync(`npx declastruct apply --plan ${planFile}`, {...});

    // regenerate plan (should show no changes if idempotent)
    execSync(`npx declastruct plan --wish ${resourcesFile} --into ${planFile}`, {...});

    // second apply (should succeed as no-op)
    execSync(`npx declastruct apply --plan ${planFile}`, {...});
  });
});
```

proves:
- operations are safe to retry
- no side effects from repeated execution
- plan accurately reflects current vs desired state

```typescript
// src/contract/sdks/.test/assets/resources.acceptance.ts
import { getDeclastructCloudflareProvider } from '../../../../../dist/contract/sdks';

export const getProviders = async () => [
```

enables:
- testing the full workflow even with empty resources
- easy extension when real zones are available
- imports from dist/ to simulate real usage

### 5. test directory structure

```
src/
├── .test/                              # shared test utilities
│   ├── getMockedCloudflareApiContext.ts  # mocked context for unit tests
│   └── getSampleCloudflareApiContext.ts  # real context for integration tests
├── contract/sdks/
│   ├── .test/
│   │   ├── .temp/                      # gitignored temp files per run
│   │   └── assets/
│   │       └── resources.acceptance.ts  # acceptance test resources
│   └── declastruct.acceptance.test.ts   # acceptance tests
├── domain.operations/
│   └── domainZone/
│       ├── getAllDomainZones.ts
│       ├── getAllDomainZones.test.ts           # unit tests
│       └── domainZone.integration.test.ts      # integration tests
```

patterns:
- `.test/` for test utilities (consistent naming)
- `.temp/` for ephemeral test artifacts (gitignored)
- tests collocated with code they test
- integration tests can span multiple operations (domainZone.integration.test.ts)

#### 6.3 rapid feedback enablers

1. **parallel test execution**: `maxWorkers: '50%'` in jest config
2. **incremental testing**: `--changedSince=main` for local dev
3. **fail-fast credential validation**: throw in env setup if missing
4. **fail-fast precondition checks**: throw with explicit messages if test resources missing
5. **self-provisioning tests**: tests should provision their own resources when possible
6. **runInBand for acceptance**: `--runInBand` prevents race conditions in cli tests
