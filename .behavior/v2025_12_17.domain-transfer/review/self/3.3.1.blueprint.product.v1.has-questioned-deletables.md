### sdk utilities

| utility | deletable? | rationale |
|---------|------------|-----------|
| getCloudflareClient | questioned | extracts client from context - one-liner utility |
| createCloudflareClient | questioned | creates new client - used in tests and exported in public SDK |

**question**: are getCloudflareClient/createCloudflareClient needed?

**analysis**:
- getCloudflareClient: extracts `context.cloudflare.client` - trivial wrapper
- createCloudflareClient: creates `new Cloudflare({apiToken})` - trivial wrapper
- both exported as public SDK utilities
- createCloudflareClient used in integration tests

**verdict**: keep as public SDK utilities for consumer convenience. the provider creates its own client directly, so these don't add complexity to core logic. if we deleted them, SDK consumers would have to import Cloudflare directly, which is fine but less ergonomic.

## simplification opportunities

### considered

1. **inline expandZoneRef into setDomainDnsRecord/getOneDomainDnsRecord?**
   - rejected: shared logic, tested separately, cleaner to keep separate

2. **merge cast functions into get operations?**
   - rejected: single responsibility - cast handles shape translation, get handles api call

3. **remove WhoisRecord as separate domain object, inline into setDomainRegistration?**
   - rejected: WhoisRecord has its own structure from api, cleaner to model explicitly

4. **remove supplementary domain objects (Settings, Contact, TransferIn)?**
   - rejected: these map real api response structures, needed for completeness

### result

no simplification opportunities that don't sacrifice clarity or correctness.
