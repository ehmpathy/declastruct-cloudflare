## anti-pattern 1: raw api calls outside operations

### what to avoid

direct calls to `client.zones.create()` outside domain operations.

### why it's bad

- bypasses validation, cast functions, error wrap
- creates inconsistent behavior paths
- makes tests harder (multiple code paths to test)
- log and observability gaps

### how the blueprint avoids it

all api interaction flows through domain operations:
- `setDomainZone` → `client.zones.create()` or `client.zones.update()`
- `getOneDomainZone` → `client.zones.list()`
- operations handle cast, validation, error wrap

### verification

search codebase for direct `client.` calls outside `domain.operations/`. should find none except in operation implementations.

## anti-pattern 2: shared mutable state

### what to avoid

global state that operations mutate, such as:
- module-level cache of zone ids
- singleton cloudflare client
- shared context mutation

### why it's bad

- race conditions between concurrent operations
- test pollution between test cases
- hard to reason about state at any point in time

### how the blueprint avoids it

- context is passed explicitly to each operation
- no module-level state
- cloudflare client created fresh per provider init (or per test)
- operations are pure functions of (input, context)

### verification

grep for `let` or `var` at module level. should find none in domain operations.

---

## anti-pattern 3: implicit dependencies between operations

### what to avoid

operations that assume other operations have run:
- `setDomainDnsRecord` assumes zone was created earlier
- `delDomainZone` assumes all records deleted

### why it's bad

- breaks when operations run out of order
- caller must track and coordinate
- hard to test in isolation

### how the blueprint avoids it

- `setDomainDnsRecord` calls `expandZoneRef` which looks up zone; fails clearly if absent
- `delDomainZone` can delegate to declastruct for dependency order
- each operation is self-contained with explicit dependencies via context

### verification

operations should not read from module-level state or assume prior operation state.

## anti-pattern 4: leaky abstractions in domain objects

### what to avoid

domain objects that expose cloudflare api internals:
- `DeclaredCloudflareDomainZone` with `account_id` as required field
- dns records with cloudflare-internal `zone_id` instead of `zone: RefByUnique`

### why it's bad

- couples user code to cloudflare api shape
- changes to api require changes to user declarations
- obscures domain concepts with implementation details

### how the blueprint avoids it

- domain objects expose clean interfaces (`zone: RefByUnique`, not `zone_id: string`)
- account_id is context, not per-resource
- cast functions translate api shapes to domain shapes

### verification

domain object interfaces should have no cloudflare-specific internal ids (except primary key `id`).

## anti-pattern 5: swallow errors silently

### what to avoid

```ts
try {
  await client.zones.delete(zoneId);
} catch (error) {
  // silently ignore
}
```

### why it's bad

- hides real failures (permission issues, network errors)
- makes debug impossible
- can leave system in inconsistent state

### how the blueprint avoids it

- operations propagate errors with context
- idempotent operations (like delete) return success for "not found" but throw for other errors
- BadRequestError used for actionable user errors
- UnexpectedCodePathError for internal invariant violations

### verification

search for empty catch blocks. should find none.

## anti-pattern 6: overload operation semantics

### what to avoid

single operation that does too much:
```ts
setDomainZone({ create?: Zone; update?: Zone; delete?: { id: string } })
```

### why it's bad

- unclear intent at call site
- complex branch logic inside
- harder to test each code path
- violates single responsibility

### how the blueprint avoids it

- separate operations: `setDomainZone` (create/update), `delDomainZone` (delete)
- findsert vs upsert via `{ findsert }` | `{ upsert }` discriminated union
- each operation has clear, focused responsibility

### verification

operation signatures should have PickOne discriminated union or single clear intent.

## anti-pattern 7: duplicate domain logic in dao and operation

### what to avoid

```ts
// in dao
const dao = {
  findsert: async (zone) => {
    const found = await getOneZone({ by: { unique: zone } });
    if (found) return found;
    return createZone(zone); // duplicate logic
  }
};
```

### why it's bad

- logic in two places; can drift
- hard to test consistently
- changes require updates in multiple files

### how the blueprint avoids it

- daos delegate to operations; no logic in daos
- `genDeclastructDao` pattern creates thin adapters
- all logic lives in domain operations

### verification

dao files should be thin wrappers with no business logic.

## anti-pattern 8: test doubles that don't match real behavior

### what to avoid

mock cloudflare client that returns static data without simulate real api behavior:
```ts
const mockClient = {
  zones: {
    list: () => [{ id: '123', name: 'test.com' }], // always returns same data
  }
};
```

### why it's bad

- tests pass but production fails
- mocks drift from real api
- false confidence in test coverage

### how the blueprint avoids it

- integration tests use real cloudflare api
- unit tests test domain logic, not api interaction
- cast functions tested with snapshot tests against real api response shapes

### verification

unit tests should not mock cloudflare client. integration tests should hit real api.

## anti-pattern 9: hardcoded constants in operations

### what to avoid

```ts
const TTL_MIN = 60; // hardcoded in operation
```

### why it's bad

- constants scattered across files
- hard to update when cloudflare changes limits
- no single source of truth

### how the blueprint avoids it

- constants defined in domain objects or dedicated constants file
- operations import from single source
- easier to update and test

### verification

search for magic numbers in operations. should reference constants.

## anti-pattern 10: mixed sync and async patterns

### what to avoid

```ts
const zone = getZoneSync(name); // blocks
await setRecord({ zone });
```

### why it's bad

- blocks event loop
- inconsistent patterns confuse developers
- sync operations can't handle api calls

### how the blueprint avoids it

- all operations are async
- `await` consistently used
- no sync api wrappers

### verification

all domain operations return Promise.

## anti-pattern 11: context shape drift

### what to avoid

operations that expect different context shapes:
```ts
// operation A
const opA = (input, context: { client: Cloudflare; log: Log }) => ...

// operation B
const opB = (input, context: { cf: Cloudflare; logger: Logger }) => ...
```

### why it's bad

- callers must adapt context per operation
- no single context type
- test requires different context shapes

### how the blueprint avoids it

- single `ContextCloudflareApi` type
- all operations expect same context shape
- context created once in provider

### verification

all operation context types should reference `ContextCloudflareApi` or compatible shape.

---

## anti-pattern 12: premature abstraction

### what to avoid

```ts
// "generic" operation that handles "any" resource
const setResource = <T>(resource: T, type: 'zone' | 'record') => {
  if (type === 'zone') { ... }
  if (type === 'record') { ... }
}
```

### why it's bad

- abstraction doesn't fit domain model
- switch statements grow with each new type
- harder to understand than specific operations

### how the blueprint avoids it

- separate operations per resource type
- no generic "resource" abstraction
- each operation handles one domain concept

### verification

no generic "Resource" type or "setResource" operation.

---

## anti-pattern 13: barrel exports that hide dependencies

### what to avoid

```ts
// index.ts
export * from './getOneZone';
export * from './setZone';
export * from './delZone';
// ... 50 more exports
```

### why it's bad

- import from index loads all modules
- circular dependency risk
- hard to tree-shake
- hides what's actually used

### how the blueprint avoids it

- direct imports from specific files
- no barrel exports in domain.operations
- only contract/sdks has public index.ts

### verification

search for `export *` in domain operations. should find none.

| state | why it's bad | how to detect | how to fix |
|-------|--------------|--------------|-----------|

## issue found: no idempotency key in zone create

from `setDomainZone.ts`:
```ts
const created = await client.zones.create({
  account: { id: accountId },
  name: zoneDesired.name,
  type: zoneDesired.type,
});
```

**issue**: no idempotency key passed. race condition risk (failure mode #3 from premortem).

**recommendation**: add idempotency key derived from unique key hash:
```ts
const created = await client.zones.create({
  account: { id: accountId },
  name: zoneDesired.name,
  type: zoneDesired.type,
}, {
  idempotencyKey: toHashSha256Sync([zoneDesired.name, zoneDesired.type].join(';')),
});
```

**status**: add to execution phase fix list.
