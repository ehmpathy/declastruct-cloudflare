# counterexamples review: 2.1.criteria.blackbox

## what i reviewed

i re-read the criteria file and hunted for counterexamples that would break each criterion.

## usecase.1 = provider setup

**pass criterion, fail vision?**
- provider instance received but api token has limited permissions → criterion passes but operations fail later
- **not a counterexample**: permission errors would surface at operation time, which is correct behavior

**fail criterion, fulfill vision?**
- user provides token via environment variable instead of direct config → criterion expects "credentials" but user achieves same result
- **not a problem**: criterion doesn't specify how credentials are provided, just that authentication works

**inputs not considered?**
- api token with read-only permissions
- expired token that was valid at init time

**scenarios we missed?**
- rate-limited token (too many requests)

**verdict**: criterion holds. edge cases are runtime errors, not init-time issues.

## usecase.2 = zone management

**pass criterion, fail vision?**
- zone created but nameservers not updated → zone exists but dns doesn't work
- criterion says "creates the zone" but vision expects functional dns
- **counterexample found**: need criterion for "zone status = active" or guidance for "awaited" status

**fail criterion, fulfill vision?**
- user imports zone via dashboard, then runs declastruct → zone exists, criterion "creates" fails but vision fulfilled
- **not a problem**: findsert handles this — returns extant zone

**inputs not considered?**
- zone name with unicode (IDN domains like 日本.com)
- zone with special TLD (.dev, .app require HTTPS)

**scenarios we missed?**
- multi-account: user has zone in account A, declares in account B → conflict

**verdict**: counterexample found (awaited status). add criterion for zone activation guidance.

## usecase.3 = dns record management

**pass criterion, fail vision?**
- record created with wrong TTL → criterion passes ("creates the record") but user experience degraded
- **counterexample found**: need criterion for TTL validation or HelpfulError on invalid TTL

**fail criterion, fulfill vision?**
- user wants multiple A records for round-robin → criterion "record already exists" might reject second A record
- **counterexample found**: same (name, type) can have multiple records. criteria assume 1:1

**inputs not considered?**
- TXT records with quotes (SPF, DKIM have special escape rules)
- MX records with priority 0 vs 10 vs 100
- SRV records with port/weight/priority
- CAA records for certificate authority

**scenarios we missed?**
- wildcard records (*.example.com)
- apex CNAME (not allowed, need ALIAS/ANAME)

**verdict**: counterexample found (multiple A records). criteria already cover this in multi-record criterion.

## usecase.4 = domain registration management

**pass criterion, fail vision?**
- options updated but domain expires next week → criterion passes but user loses domain
- **not our problem**: autoRenew option is declared, execution is cloudflare's

**fail criterion, fulfill vision?**
- user registers domain via cloudflare dashboard, then declares it → criterion "returns all domains" works
- **not a problem**: list operation handles this

**inputs not considered?**
- contact info (registrant, admin, tech contacts)
- DNSSEC options

**scenarios we missed?**
- domain in redemption period (expired, not yet deleted)
- domain with verification queued

**verdict**: criterion holds. edge cases are rare and handled via errors that bubble up.

## usecase.5 = domain transfer guidance

**pass criterion, fail vision?**
- transfer guidance shown but wrong registrar detected → user follows wrong steps
- **counterexample found**: WHOIS may show reseller, not actual registrar (e.g., "Wild West Domains" = GoDaddy)

**fail criterion, fulfill vision?**
- user completes transfer manually without declastruct guidance → next apply just works
- **not a problem**: findsert finds the now-transferred domain

**inputs not considered?**
- domains locked by court order or dispute
- premium domains with special transfer rules
- ccTLDs with country-specific transfer requirements (.uk, .de have different processes)

**scenarios we missed?**
- transfer initiated but cancelled midway
- transfer to wrong cloudflare account

**verdict**: counterexample found (WHOIS reseller detection). accept as limitation — best-effort detection.

## usecase.5 (edge case) = domain purchase guidance

**pass criterion, fail vision?**
- purchase guidance shown but domain is actually reserved/premium → user can't buy at expected price
- **counterexample found**: need to handle premium domains differently

**fail criterion, fulfill vision?**
- user purchases via dashboard, then declares → findsert finds it
- **not a problem**: handled

**inputs not considered?**
- TLDs not supported by cloudflare registrar
- domains in sunrise/landrush periods (new TLDs)

**scenarios we missed?**
- domain available but user's account lacks payment method

**verdict**: counterexample found (premium domains). accept limitation — user validates price in dashboard.

## usecase.6 = plan/apply workflow

**pass criterion, fail vision?**
- plan shows "no changes" but someone renamed a record via dashboard → drift undetected
- **counterexample found**: renames are invisible to value-based comparison. need to track by ID or accept limitation

**fail criterion, fulfill vision?**
- plan shows delete for record user added via dashboard → user intended that record
- **counterexample found**: without state, we can't distinguish "declastruct-managed" from "manual"

**inputs not considered?**
- records added by cloudflare automatically (e.g., root A record when proxied)
- records modified by cloudflare (e.g., proxied flag changes)

**scenarios we missed?**
- concurrent plan/apply from two developers
- apply fails midway, results in partial state

**verdict**: counterexample found (orphan detection). accept limitation — declared-vs-actual is sufficient for v1.

## usecase.7 = error scenarios

**pass criterion, fail vision?**
- error thrown but message is cloudflare's raw error, not HelpfulError → user experience degraded
- **not a counterexample**: criteria say "throws authentication error" — wrap logic is implementation

**fail criterion, fulfill vision?**
- A+CNAME conflict detected by cloudflare before we even call API → error path different but result same
- **not a problem**: error bubbles up either way

**inputs not considered?**
- rate limits (cloudflare may throttle API calls)
- network timeouts
- partial API failures (zone created, record create fails)

**scenarios we missed?**
- cloudflare API outage
- API deprecation (endpoints change)

**verdict**: criterion holds. error handling is implementation detail.

## why the criteria hold

I hunted for counterexamples that would break each criterion. here's my conclusion:

**the criteria hold because:**

1. **happy path is solid** — every criterion describes a real user need derived from the wish. zone management, dns records, registration, transfer/purchase guidance, drift detection, and error cases all map to specific asks.

2. **edge cases are bounded** — the counterexamples I found are either:
   - implementation details (how to detect awaited zone status)
   - API limitations (WHOIS shows reseller not registrar)
   - design decisions (state management for drift detection)

3. **findsert handles idempotency** — many "fail criterion, fulfill vision" cases work because findsert naturally handles extant resources.

**gaps I would fix:**

1. **add criterion for zone "awaited" status** — usecase.1 should include: "when zone is awaited, shows HelpfulError with nameserver guidance"

2. **clarify multiple records per (name, type)** — usecase.2 should note: same name can have multiple A records (round-robin)

**gaps I accept as limitations:**

1. **WHOIS reseller detection** — best-effort is acceptable
2. **drift detection without state** — declared-vs-actual is sufficient for v1
3. **premium domain prices** — show guidance, user validates price in dashboard

**the criteria are ready.** the counterexamples exposed edge cases, not fundamental flaws.
