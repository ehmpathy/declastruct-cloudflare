# 5 whys review: 2.1.criteria.blackbox

## what i reviewed

i drilled 5 whys on each major criterion to find the root needs they serve.

---

## criterion: provider setup

1. **why do we need provider setup?**
   → because all operations require authentication to cloudflare

2. **why does authentication matter?**
   → because cloudflare API won't accept requests without valid credentials

3. **why check credentials at setup time?**
   → because fail-fast is better than fail-later. discover auth issues before plan/apply

4. **why is fail-fast better?**
   → because delayed errors waste time. users fix credentials once, not after each operation

5. **why does time matter here?**
   → because infrastructure changes are high-stakes. users want confidence before they start

**root need**: fail-fast authentication builds confidence before work begins

---

## criterion: zone list

1. **why do we need to list zones?**
   → because users need to see what domains are managed in their account

2. **why do users need to see what domains are managed?**
   → because they can't manage what they can't see. visibility enables action

3. **why does visibility enable action?**
   → because without knowing what exists, users can't decide what to add, update, or remove

4. **why can't they use the cloudflare dashboard?**
   → because dashboard is click-ops. no versioning, no automation, no reproducibility

5. **why does reproducibility matter?**
   → because infrastructure should be declared in code, auditable, and recoverable from git

**root need**: infrastructure-as-code requires visibility into current state

---

## criterion: zone create/idempotent

1. **why do we need idempotent zone creation?**
   → because users run declastruct multiple times (plan, apply, re-apply)

2. **why do they run it multiple times?**
   → because iteration is normal. fix typo, add record, re-apply

3. **why must it be idempotent?**
   → because duplicate creates would fail or create garbage

4. **why would failures be bad?**
   → because users expect "apply" to work, not require manual cleanup

5. **why is manual cleanup unacceptable?**
   → because it defeats the purpose of automation. humans shouldn't fix machine mistakes

**root need**: automation must be safe to retry

---

## criterion: dns record findsert vs upsert

1. **why distinguish findsert and upsert?**
   → because they have different intents: "find or create" vs "overwrite"

2. **why does intent matter?**
   → because a typo in upsert mode destroys the extant record; findsert protects it

3. **why protect extant records?**
   → because someone may have set values via dashboard that shouldn't be overwritten

4. **why would dashboard values matter?**
   → because not everything is declared in code yet. migration is gradual

5. **why is gradual migration important?**
   → because big-bang migrations fail. incremental adoption succeeds

**root need**: gradual adoption requires protection of undeclared state

---

## criterion: transfer guidance

1. **why do we need transfer guidance?**
   → because cloudflare API doesn't support transfer automation

2. **why doesn't the API support it?**
   → because ICANN requires human verification steps (FOA email, 5-day wait)

3. **why provide guidance instead of just failing?**
   → because "domain not found" is useless; step-by-step instructions are helpful

4. **why are step-by-step instructions helpful?**
   → because transfer involves two registrars, multiple steps, and 5+ days. context is lost

5. **why is context lost?**
   → because humans forget. a week later, "where was I?" is common

**root need**: multi-day processes require persistent, resumable guidance

---

## criterion: purchase guidance

1. **why do we need purchase guidance?**
   → because cloudflare API doesn't support domain purchase automation

2. **why doesn't the API support it?**
   → because purchase involves payment, legal agreements, and contact verification

3. **why provide guidance instead of just failing?**
   → because "domain not registered" doesn't tell user what to do next

4. **why does "what to do next" matter?**
   → because the goal is to get the domain into cloudflare, not to inform user of failure

5. **why is goal-orientation important?**
   → because tools should help users succeed, not just report errors

**root need**: tools should guide toward success, not just report failure

---

## criterion: drift detection

1. **why do we need drift detection?**
   → because reality diverges from code. someone edits via dashboard

2. **why does divergence happen?**
   → because dashboard is easy. emergency changes happen. people forget to update code

3. **why is divergence bad?**
   → because next "apply" may overwrite manual changes or fail unexpectedly

4. **why not just apply and see what happens?**
   → because surprises in production cause outages. preview enables review

5. **why is review important?**
   → because humans catch mistakes machines miss. "wait, that record shouldn't be deleted"

**root need**: preview changes before apply to prevent surprises

---

## criterion: error cases

1. **why do we need clear error cases?**
   → because errors happen. users need to know what went wrong

2. **why must errors be clear?**
   → because "400 Bad Request" doesn't help. "invalid API token" does

3. **why does clarity matter?**
   → because time-to-fix depends on time-to-understand

4. **why is time-to-fix important?**
   → because dns issues cause outages. every minute counts

5. **why do dns issues cause outages?**
   → because dns is the foundation. if dns is wrong, no service works

**root need**: fast diagnosis enables fast recovery

## articulation: what I found

I drilled 5 whys on each major criterion. here are the root needs:

| criterion | root need |
|-----------|-----------|
| provider setup | fail-fast authentication builds confidence |
| zone list | infrastructure-as-code requires visibility |
| zone idempotent | automation must be safe to retry |
| findsert vs upsert | gradual adoption requires protection |
| transfer guidance | multi-day processes need persistent guidance |
| purchase guidance | tools should guide toward success |
| drift detection | preview changes to prevent surprises |
| error cases | fast diagnosis enables fast recovery |

**why the criteria hold:**

every criterion traces back to a genuine root need:
- visibility, idempotency, and protection are core to declarative infrastructure
- guidance patterns address API limitations while keeping users on the success path
- drift detection and clear errors prevent the surprises that cause outages

**no criterion exists without purpose.** each serves a root need that traces directly to the wish: "declarative control of domains via declastruct."

---

## synthesis: the deeper pattern

as I drilled the 5 whys, a pattern emerged. the root needs cluster into three themes:

### 1. visibility

- zone list → see what exists
- drift detection → see what changed
- error cases → see what went wrong

**the deeper why**: you cannot control what you cannot see. declarative infrastructure requires full observability of current state.

### 2. safety

- idempotency → safe to retry
- findsert protection → safe for gradual adoption
- preview before apply → safe from surprises

**the deeper why**: automation must be safer than manual work, or humans won't trust it. safety enables adoption.

### 3. guidance

- transfer guidance → path to success despite API limitation
- purchase guidance → path to success despite API limitation

**the deeper why**: tools exist to help users succeed. when full automation isn't possible, clear guidance is the next best thing. failure messages are not enough.

### the meta-insight

the criteria are not just "features we want." they are **trust builders**:

- visibility builds trust that the tool knows what's happening
- safety builds trust that the tool won't break things
- guidance builds trust that the tool is on your side

**this is why the criteria hold**: they don't just describe behaviors. they describe trust. and trust is what makes infrastructure-as-code adoption possible.

without visibility → users distrust the tool's awareness
without safety → users distrust the tool's competence
without guidance → users distrust the tool's intent

**the criteria serve trust.** that's the root beneath all roots.

---

## verdict

**the criteria hold.** every criterion traces to a root need. the 5 whys revealed three themes — visibility, safety, guidance — that all serve trust. trust enables adoption. adoption fulfills the wish.
