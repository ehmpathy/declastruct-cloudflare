### inverse.0.4: DnsRecord unique key change breaks declastruct state

**risk**: declastruct tracks resources by unique key. if unique key changes, extant state may reference old key shape.

**analysis**: declastruct stores state as `{ class, unique: {...} }`. old state has `{ zone, name, type }`. new unique key is `{ zone, name, type, content }`.

**check**: what happens when declastruct loads old state with new class?
- if state lookup fails: declastruct treats as new resource (double-create)
- if state lookup succeeds: may match wrong record

**mitigation**: declastruct must handle state migration or users must reset state.

**action**: add to execution phase: verify declastruct state behavior with changed unique key.

### inverse.0.5: multiple A records with same content

**risk**: user declares two identical A records (same zone, name, type, content). should this be allowed?

**analysis**: with unique key `['zone', 'name', 'type', 'content']`, two identical records would be deduplicated (same unique key = same record).

**verdict**: this is correct behavior. duplicate declarations are idempotent. no inverse concern.

## inverse.1: architectural states to avoid

### avoid: barrel exports in domain.operations/

**what to avoid**: creating index.ts files in `domainZone/`, `domainDnsRecord/`, `domainWhoisRecord/` directories.

**why**: barrel exports increase codepath variants and circular import risk.

**how blueprint avoids this**: filediff treestruct shows no index.ts files in operation directories. each operation is imported directly.

### avoid: mock-based unit tests for API operations

**what to avoid**: unit tests for getOneDomainWhoisRecord that mock the cloudflare client.

**why**: mocks hide real API behavior differences. per rule.forbid.remote-boundaries.

**how blueprint avoids this**: blueprint specifies integration test for getOneDomainWhoisRecord, unit test only for cast function.

### avoid: extraneous SDK exports

**what to avoid**: exporting internal implementation details like getOneDomainWhoisRecord to public SDK.

**why**: increases API surface area without user value. WHOIS is internal to guidance logic.

**how blueprint avoids this**: blueprint explicitly states `index.ts # no changes — whois is internal`.

## inverse.2: dependencies that would create problems

### avoid: circular dependency between zone and registration

**problem**: if setDomainRegistration imports from domainZone/, and setDomainZone imports from domainRegistration/, circular dependency.

**how blueprint avoids this**: registration calls WHOIS (separate module). zone operations don't depend on registration.

### avoid: external WHOIS service dependency

**problem**: if we used a third-party WHOIS service instead of cloudflare's API, adds complexity and potential auth issues.

**how blueprint avoids this**: uses cloudflare's built-in WHOIS endpoint (`client.intel.whois.get`).

### avoid: heavy runtime validation in cast functions

**problem**: if castIntoDeclaredCloudflareDomainWhoisRecord uses zod or similar for full validation, adds runtime overhead.

**how blueprint avoids this**: cast function does simple property extraction. domain object constructor handles validation.

## inverse.3: how this design could create tech debt

### tech debt risk: findsert without diff check

**how debt accumulates**: every set operation follows the pattern. if we don't fix findsert semantics now, all operations have the same defect.

**mitigation**: re-review fixes findsert in all extant operations, not just registration.

### tech debt risk: guidance messages hardcoded in setDomainRegistration

**how debt accumulates**: if transfer/purchase guidance needs update (cloudflare changes URLs, steps change), must edit source code.

**mitigation**: guidance is in one place (setDomainRegistration). clear treestruct format makes updates easy.

### tech debt risk: no abstraction for guidance pattern

**how debt accumulates**: if future operations need similar guidance (e.g., SSL certificates), pattern must be copied.

**mitigation**: YAGNI — we add abstraction when pattern repeats. current implementation is clear and localized.

## inverse.4: anti-patterns to steer clear of

### anti-pattern: UnexpectedCodePathError for user-actionable guidance

**what to avoid**: throw UnexpectedCodePathError when domain not found — this signals a bug.

**correct pattern**: throw BadRequestError with guidance — this signals user action required.

**how blueprint avoids this**: blueprint specifies BadRequestError with detailed guidance message.

### anti-pattern: optional inputs in internal contracts

**what to avoid**: using `?` optional properties in operation inputs like `setDomainRegistration({ ...opts })`.

**correct pattern**: explicit required properties with null for optional values.

**how blueprint avoids this**: blueprint follows PickOne<{ findsert; upsert }> pattern with explicit input shapes.

### anti-pattern: gerunds in variable names

**what to avoid**: names like `existingZone`, `missingRecord`, `pendingTransfer`.

**correct pattern**: `zoneFound`, `recordAbsent`, `transferQueued`.

**how blueprint avoids this**: blueprint uses correct terminology (`regFound`, `domainWhoisRecordFound`).

### anti-pattern: forbidden term "resolve"

**what to avoid**: function names like `resolveZoneRef` that use overloaded terms.

**correct pattern**: use precise terms like `expandZoneRef`, `lookupZone`, `deriveZoneId`.

**how blueprint avoids this**: blueprint uses `expandZoneRef` — extant code has `resolveZoneRef` which must be renamed.
