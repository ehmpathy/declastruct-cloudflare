## decision 1: dns record unique key includes content

### the decision

unique key = `['zone', 'name', 'type', 'content']`

### 5 whys

1. **why include content in the unique key?**
   → because cloudflare allows multiple records with the same (zone, name, type) but different content

2. **why does cloudflare allow this?**
   → because dns load distribution requires multiple A records pointing to different IPs for the same hostname

3. **why do we need to model this?**
   → because the criteria explicitly requires: "given multiple A records for load distribution are declared, when setDomainDnsRecord is called for each, then all records are created"

4. **why does declastruct care about unique keys?**
   → because findsert semantics depend on correct unique key: find by unique, insert if absent, return if present

5. **why is this the root reason?**
   → because declastruct's idempotency contract depends on correct identification of "sameness" — without content in key, two distinct A records would be treated as one, breaking load distribution

### root insight

the unique key defines identity. dns identity requires content because the same name can return multiple values intentionally.

## decision 2: zone referenced by unique (name) not primary (id)

### the decision

dns records reference zone via `RefByUnique<typeof DeclaredCloudflareDomainZone>` = `{ name: string }`

### 5 whys

1. **why reference zone by name instead of id?**
   → because users declare zones by name (the domain), not by cloudflare's internal id

2. **why don't users know the zone id?**
   → because the id is assigned by cloudflare at creation time; it doesn't exist until the zone is created

3. **why does this matter for declarations?**
   → because users declare desired state before resources exist; references must use natural keys

4. **why are natural keys better for declarations?**
   → because natural keys are human-readable and version-controllable ("mhember.com" vs "abc123xyz")

5. **why is this the root reason?**
   → because declarative infrastructure is about expressing intent, not implementation details; natural keys express intent, artificial keys leak implementation

### root insight

declarations express desired state. artificial ids don't exist yet when you write the declaration. natural keys (the domain name) are the only stable reference.

## decision 3: registration throws guidance instead of automating

### the decision

`setDomainRegistration` throws `BadRequestError` with transfer/purchase guidance when domain not in cloudflare

### 5 whys

1. **why throw an error instead of automating registration?**
   → because cloudflare's api does not support programmatic domain registration or transfer initiation

2. **why doesn't cloudflare allow this?**
   → because domain registration involves legal contracts, payment processing, and icann compliance

3. **why provide guidance in the error?**
   → because the user still wants to register/transfer; we guide them through the manual steps

4. **why use BadRequestError specifically?**
   → because the request is valid (we understand the intent) but cannot be fulfilled automatically

5. **why is this the root reason?**
   → because declastruct's contract is to make the right thing easy; when automation is impossible, guidance is the next best thing

### root insight

api limitations are facts of life. the right response is not silent failure or cryptic errors, but actionable guidance that reduces user friction.

## decision 4: whois used to detect transfer vs purchase

### the decision

when domain not in cloudflare registrar, query cloudflare's whois api to determine if transfer or purchase guidance

### 5 whys

1. **why query whois to distinguish transfer vs purchase?**
   → because the guidance differs significantly: transfer requires auth codes, purchase requires payment

2. **why not just show generic guidance?**
   → because specific guidance reduces user confusion and errors

3. **why use cloudflare's whois endpoint?**
   → because we already have cloudflare credentials and the endpoint is available at no extra cost

4. **why is accuracy acceptable even if whois is stale?**
   → because guidance is advisory; the cloudflare dashboard is the source of truth; minor confusion is acceptable

5. **why is this the root reason?**
   → because user experience trumps perfection; 95% accuracy with specific guidance beats 100% accuracy with generic guidance

### root insight

imperfect but specific guidance is more helpful than perfect but generic guidance. users discover the truth at the dashboard anyway.

## decision 5: separate cast functions per domain object

### the decision

each domain object has a corresponding `castInto*` function that translates api response to domain object

### 5 whys

1. **why separate cast functions instead of inline in get operations?**
   → because it separates concerns: get handles api call, cast handles shape translation

2. **why does separation matter?**
   → because cast logic can be tested independently with snapshot tests

3. **why are snapshot tests valuable for casts?**
   → because they catch unexpected api response shape changes immediately

4. **why not just rely on typescript types?**
   → because api responses are runtime values; types don't protect against actual shape mismatches

5. **why is this the root reason?**
   → because api contracts are brittle; separate cast functions with snapshot tests provide early warning when cloudflare changes response shapes

### root insight

external apis change without notice. cast functions with snapshot tests are an early warning system.

## decision 6: findsert throws on diff for non-updatable properties

### the decision

`setDomainDnsRecord({ findsert })` throws `BadRequestError` when record found but differs from desired on non-updatable properties

### 5 whys

1. **why throw instead of silently returning the found record?**
   → because the user declared a specific configuration; returning something different violates their intent

2. **why not just update the record?**
   → because findsert semantics are "find or insert", not "find or insert or update"

3. **why distinguish findsert from upsert?**
   → because they express different intents: findsert says "use this if absent", upsert says "make it exactly this"

4. **why is intent expression important?**
   → because it affects idempotency guarantees: findsert preserves the first value, upsert preserves the last

5. **why is this the root reason?**
   → because declarative systems must have predictable behavior; ambiguous semantics lead to bugs and confusion

### root insight

findsert and upsert are not the same. conflating them leads to unpredictable state. throwing on diff forces explicit intent.

## decision 7: provider composes daos, daos delegate to operations

### the decision

`getDeclastructCloudflareProvider` returns a provider that exposes daos, each dao delegates to domain operations

### 5 whys

1. **why this layering?**
   → because declastruct core expects a specific dao interface; we adapt our operations to fit

2. **why have operations separate from daos?**
   → because operations are testable in isolation; daos are just adapters

3. **why does testability matter here?**
   → because operations contain business logic (e.g., whois lookup for guidance); they deserve direct tests

4. **why not just put all logic in the dao?**
   → because daos would become bloated and harder to test; operations keep logic focused

5. **why is this the root reason?**
   → because clean architecture separates interface (dao) from implementation (operations); this enables testing, reuse, and evolution

### root insight

daos are contracts. operations are implementations. separating them allows each to evolve independently.

## decision 8: expandZoneRef lookup at runtime

### the decision

`expandZoneRef` looks up zone by unique key (name) to get zone id at runtime, before dns record operations

### 5 whys

1. **why look up zone at runtime instead of requiring zone id upfront?**
   → because cloudflare dns api requires zone_id, but users declare zones by name

2. **why does cloudflare require zone_id for dns operations?**
   → because zone_id is the internal partition key; it's faster than name lookup at scale

3. **why not cache zone ids?**
   → because zone ids can change if zones are deleted and recreated; stale cache causes hard-to-debug errors

4. **why is this lookup acceptable performance-wise?**
   → because zone lookup is O(1) via cloudflare's name filter; we only pay one extra api call per operation batch

5. **why is this the root reason?**
   → because we trade a small performance cost for correctness guarantees; one extra api call vs potential bugs from stale ids

### root insight

correctness > performance for infrastructure operations. stale zone ids cause silent failures. fresh lookup guarantees correctness.

### what if we chose differently?

| alternative | consequence |
|-------------|-------------|
| require zone_id upfront | users must look up zone manually; poor dx; breaks declarative model |
| cache zone ids | stale cache causes dns records to target wrong zone; silent data corruption |
| inline zone id in dns record | couples dns record to zone lifecycle; breaks reference semantics |

## decision 9: registration dao has no delete operation

### the decision

`DeclaredCloudflareDomainRegistrationDao` has `findsert` and `upsert` but no `delete`

### 5 whys

1. **why no delete operation for registrations?**
   → because cloudflare api does not support programmatic domain deletion

2. **why doesn't cloudflare allow this?**
   → because domain registrations are icann-regulated legal contracts; they can only expire or transfer

3. **why not add a "delete" that triggers transfer-out or expiry?**
   → because these are multi-step manual processes involving another registrar or waiting a year

4. **why not at least throw guidance like we do for registration?**
   → because "delete domain" is rarely the user's intent; they usually want to transfer or let expire, both of which are explicit actions

5. **why is this the root reason?**
   → because domain registrations are fundamentally different from other resources; they represent legal ownership, not just configuration

### root insight

not all resources follow CRUD semantics. domain registrations represent legal contracts. "delete" doesn't mean the same thing. the absence of delete is intentional, not an omission.

### what if we chose differently?

| alternative | consequence |
|-------------|-------------|
| add delete that throws guidance | misleading api; implies delete is possible |
| add delete that triggers transfer-out | requires destination registrar; too complex |
| add delete that disables auto_renew | silent expiry in 1 year; dangerous |

## decision 10: phased execution approach

### the decision

execution proceeds in phases: unit tests → domain objects → domain operations → daos → integration → acceptance

### 5 whys

1. **why phase the execution?**
   → because each phase validates a layer before building on it

2. **why start with unit tests?**
   → because unit tests are fast (~1s) and catch obvious regressions immediately

3. **why review domain objects before operations?**
   → because operations depend on domain objects; wrong unique keys break everything downstream

4. **why defer integration tests?**
   → because integration tests require credentials and are slow; we want fast feedback first

5. **why is this the root reason?**
   → because the dependency graph flows: objects → operations → daos → integration → acceptance; we verify bottom-up

### root insight

verify foundations before building on them. bottom-up execution catches errors where they originate, not where they manifest.

### what if we chose differently?

| alternative | consequence |
|-------------|-------------|
| run all tests at once | slow feedback; errors manifest far from cause |
| start with integration | wasted time when unit tests would have caught it |
| skip domain object review | unique key bugs discovered late in integration |

## decision 11: domain objects as separate files from operations

### the decision

each domain object (Zone, DnsRecord, Registration, etc.) is in its own file under `domain.objects/`

### 5 whys

1. **why separate files for domain objects?**
   → because domain objects are foundational; they deserve dedicated attention and tests

2. **why not inline domain objects in operations?**
   → because multiple operations share the same domain objects; duplication would cause drift

3. **why a dedicated `domain.objects/` folder?**
   → because it signals these are domain concepts, not implementation details

4. **why test domain objects separately?**
   → because domain object validation (unique keys, nested declarations) is critical and deserves focused tests

5. **why is this the root reason?**
   → because domain-driven design treats domain objects as first-class citizens; they define the ubiquitous language

### root insight

domain objects define the vocabulary. vocabulary deserves its own namespace. mixing vocabulary with implementation blurs the language.

### what if we chose differently?

| alternative | consequence |
|-------------|-------------|
| inline in operations | domain concepts scattered; hard to find authoritative definition |
| one big domain-objects.ts | all objects in one file; merge conflicts; no focused tests |
| no domain objects, use raw types | lose validation, identity semantics, reference typing |

## decision 12: integration tests per boundary, not per operation

### the decision

integration tests are organized by boundary (domainZone, domainDnsRecord, etc.), not by individual operation

### 5 whys

1. **why organize by boundary instead of operation?**
   → because boundaries represent external contracts; operations are internal

2. **why does this matter for test organization?**
   → because boundary tests verify the full round-trip: create → read → update → delete

3. **why not test each operation in isolation?**
   → because operations share state; create affects read; update affects subsequent reads

4. **why is shared state relevant?**
   → because integration tests hit real cloudflare api; we need to clean up after ourselves

5. **why is this the root reason?**
   → because integration tests are expensive (real api calls); testing boundaries is more efficient than testing operations

### root insight

integration tests verify contracts at boundaries. boundaries are where systems meet. operations are internal implementation.

### what if we chose differently?

| alternative | consequence |
|-------------|-------------|
| test per operation | duplicate api calls; 3x more expensive |
| test per domain object | misses operation-specific edge cases |
| one giant integration test | hard to debug failures; slow to run |
