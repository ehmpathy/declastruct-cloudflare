## why coverage holds: deep verification

### why domain objects hold

**DeclaredCloudflareDomainZone**:
- vision says: "container for dns, gateway to registrar"
- blueprint shows: filediff has DeclaredCloudflareDomainZone.ts, contracts table shows unique=[name], primary=[id]
- why it holds: zone is the core concept. blueprint has CRUD operations: getAll, getOne, set, del. matches vision exactly.

**DeclaredCloudflareDomainDnsRecord**:
- vision says: "individual dns records (A, AAAA, CNAME, MX, TXT, etc)"
- blueprint shows: filediff has DeclaredCloudflareDomainDnsRecord.ts, operations table has getAll, getOne, set, del
- why it holds: records are fully CRUD-able. blueprint notes fix for unique key to support multiple records with same (name, type) — this is the load distribution case from vision.

**DeclaredCloudflareDomainRegistration**:
- vision says: "registrar-level domain management"
- blueprint shows: filediff has registration domain object and operations
- why it holds: no del operation because cloudflare api doesn't support domain deletion — this is intentional, documented in vision ("some settings may not apply to all tlds").

**WhoisRecord, DnsRecordSettings, RegistrantContact, TransferIn**:
- vision says: whois for transfer detection, nested objects for complex cloudflare shapes
- blueprint shows: all files in filediff tree under domain.objects/
- why they hold: these are support domain objects that enable the core flows. TransferIn tracks transfer progress per vision.

### why operations hold

**zone operations (getAll, getOne, set, del)**:
- criteria usecase.2 says: zones must be CRUD-able with findsert/upsert/del semantics
- blueprint shows: codepath tree has setDomainZone with branches for not found, found+findsert, found+upsert
- why it holds: codepath tree explicitly shows the findsert/upsert branch logic that criteria requires.

**dns record operations (getAll, getOne, set, del)**:
- criteria usecase.3 says: records must support multiple A records for load distribution
- blueprint shows: fix noted for unique key to include content
- why it holds: blueprint explicitly identifies and schedules the fix. the filediff marks files with [~] update.

**registration operations (getAll, getOne, set)**:
- criteria usecase.4 says: update settings with autoRenew, locked, privacyProtection
- blueprint shows: codepath tree has setDomainRegistration with upsert branch
- why it holds: codepath shows update via api flow. criteria doesn't require del (cloudflare limitation).

### why guidance holds

**transfer guidance**:
- vision shows exact BadRequestError format with treestruct guidance
- blueprint shows: codepath tree has "if whois.found: throw transfer guidance"
- why it holds: codepath explicitly models the whois check and guidance flow. the getOneDomainWhoisRecord operation provides the detection.

**purchase guidance**:
- vision shows purchase guidance when domain unregistered
- blueprint shows: codepath tree has "if !whois.found: throw purchase guidance"
- why it holds: whois returns null for unregistered domains, which triggers purchase guidance branch.

### why tests hold

**unit tests**:
- criteria says: "domain object instantiation and validation"
- blueprint shows: test coverage section lists 7 domain object tests, 16 operation tests
- why it holds: every domain object and operation has collocated .test.ts file in filediff tree.

**integration tests**:
- criteria says: "against live cloudflare api"
- blueprint shows: 5 integration test files (domainZone, domainDnsRecord, domainRegistration, domainWhoisRecord, contract/sdks)
- why it holds: integration tests are blocked on credentials but files exist and are marked for review.

**acceptance tests**:
- criteria says: "declastruct plan with resources.ts"
- blueprint shows: declastruct.acceptance.test.ts and resources.acceptance.ts in filediff
- why it holds: acceptance test exists and tests the plan/apply workflow.

### why the DnsRecord unique key fix is tracked

**the issue**:
- criteria usecase.3 says: "multiple A records for load distribution are declared, then all records are created"
- current unique key: `['zone', 'name', 'type']`
- with this key: two A records with same name but different IPs would collide

**why blueprint addresses it**:
1. blueprint summary says: "fix required: DnsRecord unique key must include content"
2. filediff marks DeclaredCloudflareDomainDnsRecord.ts with [~] update
3. filediff marks getOneDomainDnsRecord.ts with [~] update
4. filediff marks setDomainDnsRecord.ts with [~] update
5. execution approach phase 1-2 covers review of domain objects and operations

the fix is not a gap — it is explicitly tracked and scheduled for execution.
