### backcompat.0.1: DnsRecord unique key state migration

**what changes**: unique key from `['zone', 'name', 'type']` to `['zone', 'name', 'type', 'content']`.

**is this a break?**: yes — declastruct stores state as `{ class, unique: {...} }`. old state files reference 3-field unique key.

**backcompat concern**: users with extant `.declastruct/` state directories will experience:
- state lookup failure → resource treated as new → double-create attempt
- or partial match → wrong record returned

**assessment**: this IS a break for users with extant state. however:
1. wish requires multiple A record support (usecase.3)
2. no way to support load distribution without `content` in unique key
3. migration path: users delete `.declastruct/` state and re-apply

**verdict**: NECESSARY BREAK — required by criteria. document migration in release notes.

## reflection: why each concern holds

### why backcompat.0.1 (unique key state) is necessary break

the break is unavoidable because:

1. **criteria demands it**: usecase.3 in blackbox criteria states "multiple A records for load distribution are declared, when setDomainDnsRecord is called for each, then all records are created"

2. **current unique key cannot distinguish**: with `['zone', 'name', 'type']`, two A records like:
   - `{ zone: 'example.com', name: 'api', type: 'A', content: '192.168.1.1' }`
   - `{ zone: 'example.com', name: 'api', type: 'A', content: '192.168.1.2' }`
   would have the same unique key → declastruct treats them as one record

3. **no backwards-compat path exists**: we cannot both support load distribution AND preserve old state shape. the state shape must change.

4. **migration is simple**: delete `.declastruct/` directory, re-run `declastruct apply`. resources in cloudflare remain intact.

**lesson**: sometimes backcompat must yield to functional correctness.

### why SDK exports (backcompat.1) holds

SDK stability is an implicit requirement because:

1. **wish says**: "follow the same patterns that are already in use in this repo"
2. **consumers exist**: anyone who uses the SDK depends on extant exports
3. **no request to break**: wish never mentioned SDK break

**lesson**: absence of break request = implicit stability requirement for public interfaces.

### why operation signatures (backcompat.2) holds

operation signature stability follows the same logic:

1. **callers depend on signatures**: a rename from `getAllDomainZones()` to `fetchAllZones()` would break callers
2. **pattern consistency**: wish explicitly requests same patterns
3. **no mention of rename**: no reason to rename operations

**lesson**: follow extant patterns unless explicitly asked to diverge.

### why error type change (backcompat.3) is not backcompat

the error type change is a bugfix, not a backcompat concern:

1. **prior behavior was wrong**: UnexpectedCodePathError signals "this should never happen" — but domain not found in registrar is expected for transfer/purchase flows
2. **BadRequestError is correct**: signals "user action needed" with guidance
3. **unlikely caller catch**: no evidence callers catch by specific error type

**lesson**: a semantic correction is a fix, not a break.

### why findsert diff check (backcompat.4) is not backcompat

the diff check is a bugfix, not a backcompat concern:

1. **prior behavior violated idempotency**: findsert that returns mismatched attributes silently is undefined behavior
2. **rule.require.idempotency**: findsert must return extant OR throw if attributes differ
3. **callers should not rely on bug**: silent mismatch was never promised behavior

**lesson**: a fix for undefined behavior is not a break.

### why internal rename (backcompat.5) is not backcompat

internal renames are invisible to consumers:

1. **not exported**: function is internal implementation detail
2. **no SDK surface**: consumers never imported it directly
3. **can rename freely**: internal names can change without notice

**lesson**: internal implementation details are not backcompat concerns.

### why additive changes (backcompat.6, 7) are not backcompat

additive changes cannot break consumers:

1. **new types don't affect old code**: consumers that don't use WhoisRecord are unaffected
2. **internal operation**: getOneDomainWhoisRecord is not exported to SDK
3. **additive is always safe**: new features don't remove old features

**lesson**: additive changes are the safest form of evolution.
