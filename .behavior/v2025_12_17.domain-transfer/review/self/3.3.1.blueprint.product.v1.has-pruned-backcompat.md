### search 2: deprecated apiKey auth

cloudflare deprecated `apiKey` auth in favor of `apiToken`. checked for any apiKey support:

```sh
grep -ri "apiKey|api_key|API_KEY" src/
# result: No matches found
```

```ts
export const createCloudflareClient = (input: {
  apiToken: string;
}): Cloudflare => {
  return new Cloudflare({
    apiToken: input.apiToken,
  });
};
```

**verdict**: only `apiToken` auth supported. no deprecated `apiKey` fallback.

## conclusion

**no backwards-compat concerns found**.

this is a greenfield implementation:
- new provider (cloudflare)
- no prior cloudflare code to maintain compat with
- aws code explicitly marked for removal (and already absent from src/)
- uses current cloudflare api (apiToken only, no deprecated apiKey)
- uses current declastruct patterns
- modern cloudflare SDK v5.2.0

### why each concern holds

| concern | why it's not an issue |
|---------|----------------------|
| AWS coexistence | wish explicitly says "we'll remove the aws resources". no AWS code in src/. |
| prior cloudflare impl | grep found no "deprecated", "legacy", "compat" markers. this is the first implementation. |
| declastruct interface | uses current `genDeclastructDao`, `findsert/upsert/del`, `RefByUnique`. no legacy patterns. |
| cloudflare api version | `createCloudflareClient` accepts only `apiToken`. no `apiKey` fallback exists. |
| domain object shapes | all domain objects are new. no prior shapes to maintain. |
| re-exports or aliases | `index.ts` exports clean paths. no alias re-exports for old import paths. |

### what would trigger backwards-compat concern

these patterns would indicate unnecessary backwards-compat:
- `// @deprecated use X instead` comments
- `apiKey?: string` optional param alongside apiToken
- re-exports like `export { NewName as OldName }`
- `legacyMode?: boolean` parameters
- version checks like `if (sdkVersion < 4) { ... }`

**none of these patterns found in codebase**.

**verdict**: no backwards-compat overhead. blueprint is clean.
