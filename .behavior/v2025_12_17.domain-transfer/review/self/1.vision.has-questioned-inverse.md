# review: 1.vision.has-questioned-inverse

## where don't we want to end up?

### anti-goal 1: users lose access to their domains

**what would this look like?**
- domain expires because autoRenew was silently disabled
- domain transferred away without user knowledge
- zone deleted and dns goes dark

**how could we get there?**
- bug in setDomainRegistration that clears autoRenew
- del operation without confirmation
- race condition between plan and apply

**mitigation:**
- findsert preserves extant values, doesn't clear them
- del is explicit operation, not implicit from absent declaration
- plan/apply model shows changes before execution

---

### anti-goal 2: users create dns records that break their services

**what would this look like?**
- A record points to wrong IP
- CNAME creates loop
- MX record has wrong priority

**how could we get there?**
- typo in content field
- misunderstand proxied config effects
- copy-paste error from another zone

**mitigation:**
- declastruct plan shows exact changes before apply
- cloudflare validates some record constraints
- vision shows clear examples

### anti-goal 3: users stuck unable to complete transfer

**what would this look like?**
- transfer initiated at wrong registrar
- auth code expired before cloudflare received it
- FOA email lost in spam

**how could we get there?**
- HelpfulError guidance unclear
- steps not in correct order
- absent prerequisite (dnssec not disabled)

**mitigation:**
- HelpfulError shows numbered steps in order
- steps include prerequisites (disable dnssec first)
- vision includes post-initiation steps (approve FOA email)

### anti-goal 4: drift undetected, infrastructure diverges from code

**what would this look like?**

- plan shows no changes when records differ
- actual state unknown

**how could we get there?**
- getAllDomainDnsRecords returns incomplete data
- comparison logic has bugs
- cache returns stale data

**mitigation:**
- declastruct plan compares declared vs actual
- no cache in read operations
- integration tests verify round-trip accuracy

### anti-goal 6: users confused by zone vs registration concepts

**what would this look like?**

- users create registration but expect dns records

**how could we get there?**

**mitigation:**

### anti-goal 7: idempotency broken, duplicate resources created

**what would this look like?**
- multiple zones with same name (if cloudflare allowed)
- duplicate dns records
- multiple transfer attempts

**how could we get there?**
- unique key mismatch
- findsert find logic wrong

**mitigation:**
- cloudflare rejects duplicate zones (external validation)

## what paths lead to outcomes we must avoid?

| path | leads to | blocked by |
|------|----------|------------|
| skip plan, direct apply | surprise changes | workflow requires plan first |
| ignore HelpfulError guidance | stuck transfer | error message is actionable |
| copy zone config for wrong domain | dns outage | zone.name is explicit per zone |
| delete zone with active traffic | outage | user must explicitly request del |
| use wrong account id | auth error | fail-fast with BadRequestError |
| transfer locked domain | transfer fails | guidance includes unlock step |

## what would make this vision fail badly?

1. **cloudflare API changes without notice** — endpoints removed or behavior changed. mitigation: integration tests detect breaks.

2. **WHOIS endpoint discontinued** — can't detect registrar. mitigation: fall back to generic guidance.

3. **domain-objects library abandoned** — no more updates. mitigation: library is ehmpathy-maintained.

4. **declastruct ecosystem abandoned** — no provider support. mitigation: declastruct is ehmpathy project.

5. **cloudflare adds purchase/transfer API** — our guidance becomes unnecessary. mitigation: this is a good problem — we'd automate it.

## what i found

### gap identified: drift detection depends on state design not yet defined

anti-goal 4 (undetected drift) has a mitigation claim: "plan compares declared vs actual"

**problem:** the vision describes drift detection as a feature but doesn't specify how state is tracked. without state:
- how do we know which resources were declared vs manually created?
- how do we detect resources that should be removed?

**analysis:** this is a design question for the blueprint phase, not a vision defect. the vision correctly describes the *goal* (detect drift). the *mechanism* belongs in blueprint.

**verdict:** gap acknowledged. flag for blueprint phase to design state strategy.

### gap identified: concurrent apply race condition

anti-goal 7 mentions "race between concurrent applies" but mitigation only says "findsert checks before create".

**problem:** if two processes run apply simultaneously:
1. both check if zone exists → not found
2. both attempt to create zone → one fails

**analysis:** this is a cloudflare-level concern. cloudflare's API would reject the second create attempt. the "failure" is actually correct behavior — only one succeeds.

**verdict:** not a gap. cloudflare API provides external validation. concurrent applies result in one success + one clear error, not silent corruption.

---

## articulation of findings

after slow, line-by-line review of the vision through the inverse lens:

**what holds:** the vision identifies the right anti-goals and provides reasonable mitigations for most. the findsert/upsert semantics, plan/apply workflow, and HelpfulError pattern all directly address inverse outcomes.

**what needs attention:** drift detection requires state management design in blueprint phase. the vision promises the capability but doesn't design the mechanism.

**overall verdict:** the vision holds. one design dependency flagged for blueprint phase (state management). no vision changes required.
