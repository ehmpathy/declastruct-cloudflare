**verdict**: assumption is correct. fix is required per criteria usecase.3.

## technical assumptions identified

### assumption.1: cloudflare WHOIS API returns registrar info

**what we assume**: `client.intel.whois.get` returns `registrar` and `registrarName` fields

**evidence**: cloudflare API docs confirm WHOIS endpoint exists at `/accounts/{account_id}/intel/whois`

**what if opposite were true**: if WHOIS API returned only domain availability (no registrar), we could not distinguish "transfer from godaddy" vs "transfer from squarespace"

**exceptions**: some TLDs may have privacy shields that hide registrar info

**simpler approach**: could skip registrar-specific messaging and show generic transfer guidance

**verdict**: assumption is valid but add fallback — if registrar is null, show generic guidance

---

### assumption.2: domain registration requires zone to be active first

**what we assume**: zone must be in `active` status before registration can be managed

**evidence**: cloudflare docs state registrar features require cloudflare as authoritative DNS

**what if opposite were true**: if registration worked before zone activation, workflow would be simpler

**counterexamples**: some registrars allow management before DNS delegation

**verdict**: assumption is cloudflare-specific constraint, not a choice — keep as is

---

### assumption.3: findsert/upsert pattern applies to registration

**what we assume**: setDomainRegistration uses findsert/upsert like other set operations

**evidence**: declastruct pattern uses PickOne<{ findsert; upsert }> consistently

**what if opposite were true**: if registration only supported upsert, findsert would fail

**counterexample**: cloudflare registration API is update-only (cannot create via API)

**verdict**: findsert for registration throws guidance error when domain not found — this is correct behavior

---

### assumption.4: BadRequestError is the right error type for guidance

**what we assume**: guidance should throw BadRequestError, not UnexpectedCodePathError

**evidence**: BadRequestError indicates "user action required" vs "bug in code"

**what if opposite were true**: UnexpectedCodePathError would suggest a bug, confusing users

**simpler approach**: could return a status object instead of throwing

**verdict**: BadRequestError is semantically correct — domain not in registrar is a bad request, guidance helps user fix it

---

### assumption.5: WHOIS lookup should be synchronous within setDomainRegistration

**what we assume**: call WHOIS API inline when registration not found

**evidence**: guidance needs WHOIS data immediately to show correct steps

**what if opposite were true**: if WHOIS was async/cached, response would be faster but possibly stale

**simpler approach**: skip WHOIS, always show both transfer and purchase options

**verdict**: synchronous WHOIS adds latency but provides accurate guidance — worth the tradeoff

---

### assumption.6: cast function is needed for WHOIS response

**what we assume**: API response should be cast into DeclaredCloudflareDomainWhoisRecord

**evidence**: cast functions enable unit testing of transform logic

**what if opposite were true**: inline transform in getOneDomainWhoisRecord

**simpler approach**: return raw API response, destructure in setDomainRegistration

**verdict**: cast function adds testability and type safety — worth the extra file

---

### assumption.7: 4 fields are sufficient for DeclaredCloudflareDomainWhoisRecord

**what we assume**: only need domain, found, registrar, registrarName

**evidence**: guidance logic only uses these fields

**what if opposite were true**: future features might need expirationDate, nameservers

**simpler approach**: current fields are sufficient for MVP

**verdict**: YAGNI — add fields when needed, not before

---

### assumption.8: integration tests are sufficient for getOneDomainWhoisRecord

**what we assume**: no unit test needed because it would require mocks

**evidence**: mocks are forbidden per rule.forbid.remote-boundaries

**what if opposite were true**: unit test with mock would catch transform bugs faster

**counterexample**: cast function has unit test, which covers transform logic

**verdict**: integration test + cast unit test provides adequate coverage

---

### assumption.9: guidance messages should use treestruct format

**what we assume**: step-by-step format with tree characters improves readability

**evidence**: treestruct is consistent with other error messages in ecosystem

**what if opposite were true**: markdown bullets might render better in some terminals

**simpler approach**: plain numbered list

**verdict**: treestruct is established pattern — consistency over preference

---

### assumption.10: accountId should be in guidance URL

**what we assume**: include `${accountId}` in cloudflare dashboard URLs

**evidence**: cloudflare dashboard URLs include account ID

**what if opposite were true**: generic URLs might work via redirect

**verdict**: specific URLs reduce user friction — keep accountId
