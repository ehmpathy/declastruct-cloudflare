## detailed articulation: why standards adherance holds

### why domain object rules adherance holds

**rule.forbid.nullable-without-reason**

the WhoisRecord domain object has:
- `domain: string` — required, identifies lookup
- `found: boolean` — required, indicates if domain is registered
- `registrar: string | null` — nullable WITH reason
- `registrarName: string | null` — nullable WITH reason

**why nullable is justified**:
- WHOIS privacy protection can hide registrar info
- domain may be registered but registrar unknown
- cloudflare API returns null for these cases

**verdict**: nullable has documented reason — not arbitrary.

---

**rule.forbid.undefined-attributes**

the WhoisRecord interface uses explicit `| null`, not `?:` optional syntax.

**why this matters**:
- undefined propagates silently
- null forces explicit handling
- callers must acknowledge nullable fields

**verdict**: no undefined attributes — all fields explicitly typed.

---

### why operation rules adherance holds

**rule.require.get-set-gen-verbs**

WHOIS operation is `getOneDomainWhoisRecord`, not:
- `fetchWhoisRecord` (forbidden: fetch)
- `lookupWhoisRecord` (forbidden: lookup)
- `queryWhoisRecord` (forbidden: query)

**why get verb is correct**:
- operation retrieves data without side effects
- follows extant pattern: `getOneDomainZone`, `getOneDomainDnsRecord`
- `getOne` indicates single-item return

**verdict**: verb follows rule.require.get-set-gen-verbs.

---

**rule.require.sync-filename-opname**

filename: `getOneDomainWhoisRecord.ts`
export: `getOneDomainWhoisRecord`

**why this matters**:
- infer procedure name from filename
- find procedures by filename search
- no mismatch between file and content

**verdict**: filename matches operation name.

---

### why error rules adherance holds

**rule.require.fail-fast**

setDomainRegistration throws BadRequestError immediately when:
1. domain not found in cloudflare registrar
2. WHOIS shows domain registered elsewhere → transfer guidance
3. WHOIS shows domain unregistered → purchase guidance

**why this is fail-fast not fail-hide**:
- error is thrown immediately, not deferred
- no silent fallback or default behavior
- guidance enables user action

**verdict**: fail-fast with HelpfulError, not fail-hide.

---

### why idempotency rules adherance holds

**rule.require.idempotency**

blueprint specifies findsert must check attribute diff:
1. lookup by unique key
2. if found, compare attributes
3. if diff, throw BadRequestError (cannot findsert with different attributes)
4. if no diff, return found

**why diff check is required**:
- without diff check, findsert returns stale data
- caller thinks their attributes were applied
- silent mismatch violates principle of least surprise

**verdict**: blueprint addresses idempotency requirement.

---

### why term rules adherance holds

**rule.forbid.term (forbidden term)**

extant code uses forbidden term in function name.
blueprint renames to `expandZoneRef`.

**why expand is correct**:
- function expands ref to full object
- ref contains `{ name }`, function returns `{ name, id }`
- "expand" precisely describes the operation

**verdict**: forbidden term replaced with precise alternative.

---

**rule.forbid.gerunds**

blueprint uses:
- `regFound` (not `existingReg`)
- `domainWhoisRecordFound` (not `matchingWhoisRecord`)
- `zoneFound` (not `loadedZone`)

**why [noun][state] pattern**:
- autocomplete groups variants: regFound, regCreated, regUpdated
- scan efficiency: noun first, state second
- no gerund confusion

**verdict**: no gerund variable names.
