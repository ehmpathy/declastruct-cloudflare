## ISSUE FOUND: dns record unique key assumption

### the assumption

the implementation assumes `[zone, name, type]` is the unique key for dns records.

### what the code has

```ts
// DeclaredCloudflareDomainDnsRecord.ts
public static unique = ['zone', 'name', 'type'] as const;
```

```ts
// getOneDomainDnsRecord.ts
for await (const r of client.dns.records.list({...})) {
  records.push(...);
}
return records[0] ?? null;  // returns only FIRST match!
```

### what the criteria requires

from 2.1.criteria.blackbox.md:
> given multiple A records for load distribution are declared
>   when setDomainDnsRecord is called for each
>     then all records are created
>       sothat round-robin load distribution works

### the discrepancy

- cloudflare allows MULTIPLE records with same (zone, name, type) - e.g., multiple A records for round-robin load balancing
- the unique key `[zone, name, type]` treats them as one entity
- getOneDomainDnsRecord returns only the first match, ignoring others
- this breaks the load distribution usecase

### what the distillation said

from 3.2.distill.domain._.v1.i1.md:
> public static unique = ['zone.name', 'name', 'type', 'content'] as const

the distillation correctly identified that `content` should be part of the unique key!

### fix needed

**option 1**: change unique key to `['zone', 'name', 'type', 'content']`
- pros: matches criteria, supports load distribution
- cons: more complex findsert logic

**option 2**: accept limitation, document that multiple records with same (name, type) are not supported
- pros: simpler
- cons: violates criteria

**recommendation**: option 1 - update unique key to include content. this must be fixed in execution phase.

## VERIFIED: registration has no delete operation

### the assumption

DeclaredCloudflareDomainRegistrationDao has no `set.delete` method.

### what do we assume here without evidence?

that cloudflare does not support programmatic domain deletion.

### is this verified?

yes. from cloudflare api docs and the registrar api:
- domains can be transferred out via `registrar.domains.transfer`
- domains can be allowed to expire by disabling auto_renew
- no `registrar.domains.delete` endpoint exists
- this matches icann regulations: domain registrations cannot be arbitrarily deleted

### what if the opposite were true?

if delete were possible, we would add `set.delete` to the dao. however, this would be an unusual registrar feature (icann requires transfer or expiry, not deletion).

### why this holds

domain registrations represent legal ownership records. unlike zones (which are just dns configuration), registrations are icann-regulated contracts. the absence of a delete operation is correct and intentional, not an oversight.

## VERIFIED: zone type includes 'secondary'

### the assumption

zone type allows 'full' | 'partial' | 'secondary'.

### what do we assume here without evidence?

that these three types are complete and sufficient.

### evidence

from cloudflare api schema (zones.create):
- `type` is required enum: `full`, `partial`, `secondary`
- no other values are accepted

### what if the opposite were true?

if cloudflare added a fourth zone type (e.g., `forwarding`), we would need to update the type union. however:
- the type comes from cloudflare's api schema
- we model what the api accepts
- breaking changes to cloudflare api would require sdk update regardless

### why this holds

we derive the type union from cloudflare's api, not from assumption. this is evidence-based, not habit-based. when cloudflare changes the api, the official cloudflare sdk types would update, and typescript would flag the discrepancy.

## VERIFIED: WHOIS for transfer detection

### the assumption

we use WHOIS api to detect if domain is registered elsewhere.

### what do we assume here without evidence?

that WHOIS data is accurate enough to provide useful guidance.

### what if WHOIS data is stale?

WHOIS data can be up to 24-48 hours stale. failure modes:
- if WHOIS shows registered but domain is actually available: user gets transfer guidance for unregistered domain (minor confusion, they'll discover at cloudflare dashboard)
- if WHOIS shows unregistered but domain is actually registered: user gets purchase guidance when they need transfer (they'll discover at cloudflare dashboard)

### could a simpler approach work?

alternatives considered:
1. **skip WHOIS, always show generic guidance**: worse ux, no registrar-specific instructions
2. **query multiple WHOIS sources**: complexity, latency, diminishing returns
3. **use cloudflare's WHOIS api only**: what we do - cloudflare likely has good cache

### why this holds

- guidance is advisory, not authoritative
- cloudflare dashboard is the source of truth
- WHOIS provides value: registrar-specific transfer instructions
- staleness risk is acceptable for advisory guidance
- no user action is taken based on WHOIS alone - they must interact with cloudflare dashboard

the tradeoff is: useful advisory guidance vs. occasional minor confusion (resolved at dashboard). we accept this.

## VERIFIED: findsert throws on diff

### the assumption

findsert operations throw BadRequestError when extant entity differs from desired.

### is this the right behavior?

yes. findsert semantics are:
- "find extant OR insert new"
- if extant has different attributes, user intent is ambiguous
- throwing forces user to be explicit: use upsert if they want to update

this matches findsert pattern used in declastruct-stripe-sdk and other declastruct providers.
