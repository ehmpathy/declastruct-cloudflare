## detailed articulation: why adherance holds

### why adherance.0.1 (DnsRecord unique key fix) holds

**criteria quote**: "given multiple A records for load distribution are declared, when setDomainDnsRecord is called for each, then all records are created"

**why this adherance is not coincidental**:

1. the vision explicitly mentions load distribution as a use case
2. the criteria makes it a testable requirement
3. the blueprint fix is the minimal change to enable it

without content in unique key:
```
record A: { zone: 'x', name: 'api', type: 'A', content: '1.1.1.1' }
record B: { zone: 'x', name: 'api', type: 'A', content: '2.2.2.2' }
```
both records have same unique key → declastruct sees them as one record → only one created

with content in unique key:
- record A unique key: `{ zone: 'x', name: 'api', type: 'A', content: '1.1.1.1' }`
- record B unique key: `{ zone: 'x', name: 'api', type: 'A', content: '2.2.2.2' }`
- both records distinct → both created

**lesson**: the blueprint fix is not an arbitrary decision — it's directly traceable to the criteria requirement.

### why vision.3 (transfer guidance) adherance holds

**vision quote**: treestruct format with `│`, `├─`, `└─` characters

**why this adherance is not coincidental**:

1. the vision shows exact message format — not a suggestion but a specification
2. the blueprint declares BadRequestError.throw with treestruct format
3. the guidance includes registrar-specific information from WHOIS

**lesson**: the blueprint adheres because the vision was specific enough to leave no room for interpretation.

### why junior deviations were caught

three deviations found in extant code:

1. **error type**: UnexpectedCodePathError (signals bug) vs BadRequestError (signals user action needed)
2. **findsert semantics**: silent mismatch (undefined behavior) vs diff check + error (defined behavior)
3. **forbidden term**: violates rule.forbid.term

**why these matter**:

- error type affects user experience — "unexpected" vs "you need to do X"
- findsert semantics affects idempotency guarantees
- forbidden terms affect codebase consistency

**lesson**: junior deviations often stem from not reading the vision/criteria carefully. the blueprint review caught them.

### why no over-engineering occurred

**wish says**: "follow the same patterns that are already in use in this repo"

**evidence of adherance**:

- all operations follow extant get/set/del pattern
- all domain objects follow DeclaredCloudflare{Resource} pattern
- all cast functions follow castIntoDeclared{Resource} pattern
- no new abstractions invented

**the one exception**: WHOIS operations are new, but they follow the same pattern as zone/record/registration.

**lesson**: adherance to wish means minimal divergence from extant patterns.
