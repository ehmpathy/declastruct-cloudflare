## question 1: what could go wrong?

### edge case 1.1: domain registered elsewhere (transfer needed)

**what could go wrong**: user declares a domain registration for a domain registered at another registrar. setDomainRegistration cannot automate the transfer.

**playtest coverage**: edgey path 1

**why it holds**: the playtest instructs foreman to run:
```sh
npm run test:unit -- setDomainRegistration.test.ts
```
and look for:
- `when('the registration does not exist and whois shows domain is registered elsewhere') then('it should throw a BadRequestError with transfer guidance')`

the test verifies the error includes step-by-step guidance for transfer at both registrars.

### edge case 1.2: domain not registered (purchase needed)

**what could go wrong**: user declares a domain registration for a domain that is not registered anywhere. setDomainRegistration cannot automate the purchase.

**playtest coverage**: edgey path 2

**why it holds**: the playtest instructs foreman to run:
```sh
npm run test:unit -- setDomainRegistration.test.ts
```
and look for:
- `when('the registration does not exist and whois shows domain is unregistered') then('it should throw a BadRequestError with purchase guidance')`

the test verifies the error includes step-by-step guidance for purchase via cloudflare dashboard.

### edge case 1.3: zone not found in lookup

**what could go wrong**: user calls getOneDomainZone with a zone that does not exist.

**playtest coverage**: happy path 1

**why it holds**: the integration test includes:
- `when('[non-extant zone id] getOneDomainZone is called') then('it should return null')`

the test verifies null is returned, not an error thrown.

### edge case 1.4: dns record not found in lookup

**what could go wrong**: user calls getOneDomainDnsRecord with a record that does not exist.

**playtest coverage**: happy path 2

**why it holds**: the integration test includes:
- `when('[non-extant record] getOneDomainDnsRecord is called') then('it should return null')`

the test verifies null is returned, not an error thrown.

### edge case 1.5: registration not found in lookup

**what could go wrong**: user calls getOneDomainRegistration with a registration that does not exist.

**playtest coverage**: happy path 3

**why it holds**: the integration test includes:
- `when('[non-extant registration] getOneDomainRegistration is called') then('it should return null')`

the test verifies null is returned, not an error thrown.

### edge case 1.6: invalid credentials

**what could go wrong**: user provides invalid cloudflare credentials.

**playtest coverage**: prerequisites section

**why it holds**: the playtest instructs foreman to run:
```sh
curl -s -X GET "https://api.cloudflare.com/client/v4/user/tokens/verify" \
  -H "Authorization: Bearer $CLOUDFLARE_API_TOKEN" | jq .success
```
expected: `true`. if credentials are invalid, this returns `false` before any tests run.

## question 2: what inputs are unusual but valid?

### edge case 2.1: multiple A records with same name (load distribution)

**unusual input**: two A records for `api.example.com` with different IP addresses. this is valid for round-robin load distribution.

**playtest coverage**: edgey path 3

**why it holds**: the playtest instructs foreman to run:
```sh
npm run test:types
```
and verifies:
- unique key for DnsRecord is `['zone', 'name', 'type', 'content']` — not just `['zone', 'name', 'type']`
- unique lookups require `content` field

the type system enforces that content is part of the unique key, so two records with same name but different content are treated as distinct.

### edge case 2.2: zone ref by name (unique) vs by id (primary)

**unusual input**: some operations accept zone by name, others by id. both are valid.

**playtest coverage**: happy path 2

**why it holds**: the integration tests include:
- `when('[t0] getAllDomainDnsRecords is called by zone id') then('...')` — id lookup
- `when('[t1] getAllDomainDnsRecords is called by zone name') then('it should expand zone and return records')` — name lookup with expansion
- `when('[t2] getOneDomainDnsRecord is called with zone name ref') then('it should expand zone and return the record')` — unique lookup with zone expansion

the tests verify both ref types work.

### edge case 2.3: TXT records with special characters

**unusual input**: TXT records may contain quotes, semicolons, and other special characters.

**playtest coverage**: happy path 2 (unit tests)

**why it holds**: the unit tests in `castIntoDeclaredCloudflareDomainDnsRecord.test.ts` include:
- `given('a TXT record') when('cast is called') then('it should correctly cast TXT record type')`

the cast function handles TXT record content correctly.

### edge case 2.4: MX records with priority

**unusual input**: MX records require a priority field.

**playtest coverage**: happy path 2 (unit tests)

**why it holds**: the unit tests in `castIntoDeclaredCloudflareDomainDnsRecord.test.ts` include:
- `given('an MX record with priority') when('cast is called') then('it should include priority field')`

the domain object and cast function support priority for MX records.

## question 3: are boundaries tested?

### boundary 3.1: empty zones list

**boundary**: account has no zones.

**playtest coverage**: happy path 1

**why it holds**: the integration test includes:
- `then('it should return an array of zones')` — works even if array is empty

the playtest expected outcome says "(may be empty)" — acknowledges this boundary.

### boundary 3.2: empty dns records list

**boundary**: zone has no dns records.

**playtest coverage**: happy path 2

**why it holds**: the integration test includes:
- `then('it should return an array of records')` — works even if array is empty

the playtest expected outcome says "(may be empty)" — acknowledges this boundary.

### boundary 3.3: empty registrations list

**boundary**: account has no domain registrations.

**playtest coverage**: happy path 3

**why it holds**: the integration test includes:
- `then('it should return an array of registrations')` — works even if array is empty

the playtest expected outcome says "(may be empty)" — acknowledges this boundary.

### boundary 3.4: findsert idempotency (record exists)

**boundary**: findsert called when record already exists with same attributes.

**playtest coverage**: unit tests

**why it holds**: the unit tests verify each domain entity:

setDomainZone.test.ts:
- `given('a findsert operation') when('the zone already exists') then('it should return the extant zone without modification')` — zone findsert idempotency

setDomainDnsRecord.test.ts:
- `given('a findsert operation') when('the record already exists') then('it should return the extant record without modification')` — record findsert idempotency

setDomainRegistration.test.ts:
- `given('a findsert operation') when('the registration exists with same attributes') then('it should return the extant registration without modification')` — registration findsert idempotency

all three entities support findsert idempotency. the tests verify no api call is made when extant matches.

### boundary 3.5: findsert conflict (record exists with different attributes)

**boundary**: findsert called when record already exists with different attributes.

**playtest coverage**: unit tests

**why it holds**: the unit tests verify:

setDomainRegistration.test.ts:
- `given('a findsert operation') when('the registration exists with different attributes') then('it should throw a BadRequestError')` — conflict detection

setDomainZone.test.ts:
- `given('a zone with mismatched id') when('extant zone id differs from expected id') then('it should throw an error')` — id mismatch detection

setDomainDnsRecord.test.ts:
- `given('a record with mismatched id') when('extant record id differs from expected id') then('it should throw an error')` — id mismatch detection

findsert throws error if attributes differ — this prevents silent data corruption. the user must use upsert to intentionally overwrite.

### boundary 3.6: delete idempotency (record absent)

**boundary**: delete called when record does not exist.

**playtest coverage**: unit tests

**why it holds**: the unit tests verify:

delDomainDnsRecord.test.ts:
- `given('a delete by primary key (id)') when('the record does not exist (not found error)') then('it should return deleted: true (idempotent)')` — idempotent on primary key
- `given('a delete by unique key (zone, name, type, content)') when('the record does not exist') then('it should return deleted: true without api call')` — idempotent on unique key

delDomainZone.test.ts:
- `given('a delete by primary key (id)') when('the zone does not exist (not found error)') then('it should return deleted: true (idempotent)')` — zone delete idempotency

delete succeeds silently when record is absent. no error is thrown. safe to retry.
