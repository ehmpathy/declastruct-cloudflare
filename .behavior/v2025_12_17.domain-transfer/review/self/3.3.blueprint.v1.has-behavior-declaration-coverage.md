## detailed articulation: why each requirement holds

### why zone management (requirement.1) holds

**vision quote**: "get/set operations to see what domains are already in cloudflare"

**blueprint provides**:
- getAllDomainZones: enumerate all zones in account
- getOneDomainZone: fetch single zone by id or name
- setDomainZone: create or update zone
- delDomainZone: remove zone

**why it holds**: blueprint provides full CRUD for zones. "see what domains" = getAllDomainZones. "add new" = setDomainZone.

### why dns record management (requirement.2) holds

**vision quote**: "manage the dns of those domains"

**blueprint provides**:
- getAllDomainDnsRecords: enumerate records for a zone
- getOneDomainDnsRecord: fetch single record
- setDomainDnsRecord: create or update record
- delDomainDnsRecord: remove record

**why it holds**: blueprint provides full CRUD for dns records. all dns management scenarios covered.

### why transfer guidance (requirement.4) holds

**vision quote**: "manage the import of domains into cloudflare, transfered out of another registrar"

**blueprint provides**:
- getOneDomainWhoisRecord: detect if domain is registered elsewhere
- setDomainRegistration: throws BadRequestError with transfer guidance when domain registered elsewhere

**why it holds**: vision says "import" which we implement as "guided transfer". cloudflare API doesn't support automated transfer. blueprint provides the maximum possible automation (detection) plus clear guidance (HelpfulError).

### why DnsRecord unique key fix (requirement.0.3) holds

**criteria quote**: "multiple A records for load distribution are declared, when setDomainDnsRecord is called for each, then all records are created"

**blueprint fix**: unique key `['zone', 'name', 'type', 'content']`

**why it holds**: with `content` in unique key, these two records are distinct:
```
{ zone: 'example.com', name: 'api', type: 'A', content: '192.168.1.1' }
{ zone: 'example.com', name: 'api', type: 'A', content: '192.168.1.2' }
```

without `content`, they would be the same unique key → only one record created. the fix enables the load distribution use case.
