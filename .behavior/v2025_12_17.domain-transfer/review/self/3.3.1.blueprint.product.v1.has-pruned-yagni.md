### DeclaredCloudflareDomainDnsRecordSettings — analysis

**what is it**: nested object for dns record advanced settings

**was it requested?**: not explicitly in vision or criteria

**actual code** (from `DeclaredCloudflareDomainDnsRecordSettings.ts`):
```ts
export interface DeclaredCloudflareDomainDnsRecordSettings {
  ipv4Only?: boolean;
  ipv6Only?: boolean;
  flattenCname?: boolean;
}
```

**field count**: 3 optional boolean fields

**could it be simpler?**: yes — could inline as `settingsIpv4Only`, `settingsIpv6Only`, `settingsFlattenCname` on DnsRecord

**why separate object holds**:
1. cloudflare api returns `settings` as nested object — matches api shape
2. settings object referenced in `DeclaredCloudflareDomainDnsRecord.nested = { settings: ... }`
3. nested declaration enables automatic hydration via domain-objects library
4. keeps main dns record interface focused on core fields

**verdict**: acceptable complexity. only 3 fields but api returns nested object. inline would diverge from api shape.

### DeclaredCloudflareDomainRegistrantContact — analysis

**what is it**: nested object for registrant contact info

**was it requested?**: not explicitly, but registration management is requested

**actual code** (from `DeclaredCloudflareDomainRegistrantContact.ts`):
```ts
export interface DeclaredCloudflareDomainRegistrantContact {
  id?: string;
  firstName: string;

  address2?: string;
  city: string;
  state: string;
  zip: string;
  country: string;
  phone: string;
  fax?: string;
  email?: string;
}
```

**field count**: 13 fields (10 required, 3 optional)

**could it be simpler?**: theoretically yes — could inline all 13 fields with `contact` prefix

**why separate object holds**:
1. 13 fields would bloat DeclaredCloudflareDomainRegistration if inline
2. cloudflare api returns contact as nested object
3. contact is a coherent domain concept (address, name, phone)
4. registration already has many fields (name, autoRenew, locked, etc)

**verdict**: required for cleanliness. 13 fields justifies separate object.

### DeclaredCloudflareDomainTransferIn — analysis

**what is it**: tracks transfer status fields

**was it requested?**: yes — vision explicitly says "transferIn status fields track progress — guidance can resume where you left off"

**actual code** (from `DeclaredCloudflareDomainTransferIn.ts`):
```ts
export interface DeclaredCloudflareDomainTransferIn {
  acceptFoa: string;
  approveTransfer: string;
  canCancelTransfer: string;
  disablePrivacy: string;
  enterAuthCode: string;
  unlockDomain: string;
}
```

**field count**: 6 status fields

**could it be simpler?**: could collapse to single `transferStatus` enum

**why separate object holds**:
1. vision explicitly requests tracking transfer progress
2. each field represents a distinct step in transfer flow
3. cloudflare api returns transfer_in as nested object
4. enables per-step status inspection ("did user accept FOA?")

**verdict**: ✅ required — explicitly mentioned in vision. enables step-by-step transfer tracking.
