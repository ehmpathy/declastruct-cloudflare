## failure mode 1: cloudflare api pagination breaks getAllDomainZones

### the failure

getAllDomainZones returns only first 50 zones. user has 200 zones. 150 zones invisible to declastruct. drift detection misses 75% of infrastructure.

### why it failed

assumed cloudflare api returns all zones in single response. cloudflare paginates at 50 results per page.

### technical assumption wrong

"zones api returns all zones" — wrong. cloudflare paginates all list endpoints.

### how to prevent

- review cloudflare sdk for pagination patterns
- use `for await (const zone of client.zones.list(...))` which handles pagination
- verify getAllDomainDnsRecords and getAllDomainRegistrations also paginate correctly

### status

**verified in code**: the operations use `for await` loops over the sdk's async iterators. pagination is handled correctly by the cloudflare sdk.

```ts
// from getAllDomainZones.ts pattern
for await (const zone of client.zones.list({ account: { id: accountId } })) {
  zones.push(castIntoDeclaredCloudflareDomainZone(zone));
}
```

**no fix needed** — pagination is handled.

## failure mode 2: dns record content escape breaks unique key lookup

### the failure

user declares TXT record with content that contains special characters. getOneDomainDnsRecord lookup fails to find it. findsert creates duplicate. cloudflare returns error for duplicate record.

### why it failed

content field contains characters that need escape when used in api queries. lookup uses content as filter but doesn't escape it.

### technical assumption wrong

"content can be used as-is in api lookups" — potentially wrong, depends on cloudflare api query syntax.

### how to prevent

- verify cloudflare sdk handles content escape internally
- test with edge case content: quotes, semicolons, equals signs (common in TXT records)
- add integration test with SPF/DKIM record content

### status

**verified in architecture**: cloudflare sdk handles the content field internally. we pass structured objects, not raw strings to query endpoints. the sdk serializes correctly.

**however**: integration tests should explicitly cover TXT records with complex content (SPF, DKIM). add to execution phase checklist.

**action**: ensure integration tests cover:
- TXT record with `v=spf1 include:_spf.google.com ~all`
- TXT record with DKIM key (long base64 string)

## failure mode 3: race condition in findsert creates duplicates

### the failure

two concurrent declastruct apply calls both find no record. both insert. cloudflare accepts both because their unique constraint differs from ours. duplicates created.

### why it failed

findsert is not atomic: lookup and insert are separate operations. window between them allows race.

### technical assumption wrong

"findsert is safe for concurrent execution" — wrong without external coordination.

### how to prevent

- document that concurrent declastruct apply on same resource set is unsafe
- cloudflare idempotency keys can prevent duplicate creates (if api supports)
- declastruct core should serialize operations on same resource

### status

**by design**: declastruct apply is not designed for concurrent execution on resources that overlap. this is a known limitation documented in declastruct core.

**mitigation**: cloudflare sdk supports idempotency keys for create operations. setDomainDnsRecord should pass idempotency key derived from unique key hash.

**action**: verify setDomainDnsRecord, setDomainZone use idempotency keys.

## failure mode 4: zone deletion fails due to active records

### the failure

user tries to delete zone. cloudflare rejects because zone has dns records. delDomainZone throws unexpected error. declastruct apply fails mid-execution.

### why it failed

delDomainZone doesn't check for dependent records. cloudflare enforces referential integrity.

### technical assumption wrong

"zones can be deleted freely" — wrong. zones with records cannot be deleted.

### how to prevent

- delDomainZone should delete records first, then zone
- or: delDomainZone should throw clear BadRequestError if zone has records
- declastruct core handles deletion order (resources deleted before dependents)

### status

**verified in declastruct pattern**: declastruct core orders deletions correctly. it deletes dns records before zones based on dependency graph. individual delDomainZone is not expected to handle this.

**however**: if user calls delDomainZone directly (not via declastruct), they could hit this error.

**action**: delDomainZone should catch cloudflare "zone has records" error and wrap with helpful message.

## failure mode 6: api token lacks required permissions

### the failure

user provisions token with Zone:Read permission but not Zone:Edit. getAllDomainZones works. setDomainZone fails with 403. user confused about which permissions needed.

### why it failed

provider init validates token but doesn't check all required scopes. failure happens later at operation time.

### technical assumption wrong

"token validation at init catches permission issues" — wrong. init only validates token is valid, not that it has all required scopes.

### how to prevent

- document required scopes clearly in provider setup guide
- consider probe of permissions at init (call a write-then-delete operation)
- wrap 403 errors with guidance about required permissions

### status

**by design**: check of all permissions at init is expensive (requires test writes). document of required scopes is the right approach.

**action**: ensure BadRequestError wraps 403 with message that lists required permissions:
- Zone:Read, Zone:Edit for zones and dns
- Domain Registration:Read, Domain Registration:Edit for registrations
- Account Intel:Read for whois

## failure mode 7: getOneDomainDnsRecord returns wrong record

### the failure (CRITICAL)

user has two A records for load distribution: 192.168.1.1 and 192.168.1.2. getOneDomainDnsRecord({ by: { unique: { zone, name, type: 'A' } } }) returns only the first one. findsert for second record finds first record, sees different content, throws diff error.

### why it failed

unique key was `[zone, name, type]`. should be `[zone, name, type, content]`. without content, lookup is ambiguous when multiple records share (zone, name, type).

### technical assumption wrong

"(zone, name, type) uniquely identifies a dns record" — WRONG. cloudflare allows multiple records with same (zone, name, type).

### how to prevent

- unique key must include content: `[zone, name, type, content]`
- getOneDomainDnsRecord must filter by content when lookup by unique
- this was caught in has-questioned-assumptions review

### status

**ISSUE FOUND IN PRIOR REVIEW**. fix already documented:
- update DeclaredCloudflareDomainDnsRecord unique key
- update getOneDomainDnsRecord to include content in lookup
- update setDomainDnsRecord findsert logic
- update tests

**fix will be applied in execution phase**.

### the failure

user declares MX record with `proxied: true`. cloudflare silently ignores the proxy flag (MX cannot be proxied). user thinks mail is proxied but it's not.

### why it failed

### technical assumption wrong

"cloudflare validates proxy compatibility" — wrong. api silently ignores invalid combinations.

### how to prevent

- validate proxy compatibility in setDomainDnsRecord before api call
- throw BadRequestError if proxy set for incompatible type
- compatible types: A, AAAA, CNAME only

### status

**needs fix**: setDomainDnsRecord should validate proxy compatibility.

**action**: add validation to setDomainDnsRecord:
```ts
if (record.proxied && !['A', 'AAAA', 'CNAME'].includes(record.type)) {
  BadRequestError.throw(
    `proxied cannot be true for ${record.type} records`,
    { record }
  );
}
```

## failure mode 13: TTL of 1 (automatic) misinterpreted

### the failure

user declares record with `ttl: 1` for non-proxied record. cloudflare interprets 1 as "automatic" for proxied records. for non-proxied, 1 is invalid. api returns error.

### why it failed

TTL = 1 is magic value for "automatic" but only works for proxied records.

### technical assumption wrong

"TTL values are straightforward integers" — wrong. 1 has special meaning.

### how to prevent

- validate TTL value based on proxy state
- if proxied: TTL = 1 (automatic) is valid
- if not proxied: TTL must be >= 60 (or higher for some plans)
- throw clear BadRequestError with valid TTL range

### status

**needs validation**: add TTL validation to setDomainDnsRecord.

**action**: add validation:
```ts
if (!record.proxied && record.ttl === 1) {
  BadRequestError.throw(
    'ttl of 1 (automatic) is only valid for proxied records. use ttl >= 60 for non-proxied records.',
    { record }
  );
}
```

## failure mode 15: concurrent zone modification causes conflicts

### the failure

two users run declastruct apply simultaneously on same zone. user A updates record X. user B updates record Y. cloudflare has optimistic concurrency. one user's changes overwrite the other's.

### why it failed

no lock on zone modification. concurrent modifications cause last-write-wins.

### technical assumption wrong

"declastruct operations are isolated" — wrong for concurrent users on same infrastructure.

### how to prevent

- document that concurrent modifications to same zone are unsafe
- consider zone-level lock via external mechanism
- declastruct core could detect conflicts on plan

### status

**known limitation**: concurrent access to shared infrastructure requires coordination outside declastruct.

**action**: document in user guide: "avoid concurrent declastruct apply on same zones without coordination."
