# requirements review: 2.1.criteria.blackbox

## what i reviewed

i re-read the criteria file and matched each usecase against the wish and vision.

---

## usecase.1 = provider setup

**who said this was needed?** foundation — every operation requires authentication

**what if removed?** users couldn't connect to cloudflare. all operations would fail without a provider.

**scope right?** yes — two criteria cover success and failure paths

**simpler test?** no — these are atomic boundary cases

**verdict**: required

---

## usecase.2 = zone management

**who said this was needed?** the wish explicitly asks for "get/set operations to see what domains are already in cloudflare"

**what if removed?** users couldn't manage zones via code — core to the vision

**scope right?** yes — covers episode (declare/plan/apply) + exchanges (get/set/del) + idempotency

**simpler test?** no — these are atomic operations already

**verdict**: required

---

## usecase.3 = dns record management

**who said this was needed?** the wish asks for "get/set operations to manage the dns of those domains"

**what if removed?** users couldn't manage dns records — the primary use case

**scope right?** yes — covers CRUD with findsert/upsert semantics + multi-record support

**simpler test?** the findsert vs upsert distinction is essential to the declastruct pattern. the multi-record criterion for load distribution is worth keeping because the vision mentions it.

**verdict**: required

---

## usecase.4 = domain registration management

**who said this was needed?** the wish asks for operations to "register new ones"

**what if removed?** users couldn't manage registration settings like autoRenew

**scope right?** yes — minimal, focused on settings. note: only covers domains already in cloudflare.

**simpler test?** already minimal

**verdict**: required

---

## usecase.5 = domain transfer guidance

**who said this was needed?** the wish explicitly asks for "get/set operations to manage the import of domains into cloudflare, transferred out of another registrar"

**what if removed?** users would get cryptic errors when domains need transfer — core differentiator

**scope right?** yes — covers the episode (detect → guide → complete) and edge case (unregistered domain)

**simpler test?** already minimal

**verdict**: required

---

## usecase.6 = the plan/apply workflow

**who said this was needed?** the wish says "declastruct declarative control" — plan/apply is the declastruct pattern

**what if removed?** users couldn't preview changes before apply. defeats declarative intent.

**scope right?** yes — covers declare/plan/apply + idempotency + drift detection

**simpler test?** already minimal

**verdict**: required — core to declastruct pattern

---

## usecase.7 = error scenarios

**who said this was needed?** implicit in pit-of-success approach — users need clear errors

**what if removed?** users would face cryptic errors and not know what to fix

**scope right?** yes — covers invalid input, api errors, and auth expiry

**simpler test?** could consolidate error cases into other usecases, but separate visibility is useful

**verdict**: required

---

## boundary conditions

**who said this was needed?** implicit from dns/cloudflare domain knowledge — these are real constraints

**what if removed?** users would discover limits at runtime with unclear errors

**scope right?** yes — captures key constraints without exhaustive enumeration

**verdict**: keep for context, serves as documentation

all 7 usecases trace back to the wish or derive from first principles:
