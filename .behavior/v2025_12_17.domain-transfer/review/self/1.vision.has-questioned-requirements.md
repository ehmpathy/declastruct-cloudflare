# review: 1.vision.has-questioned-requirements

## requirement 1: zone management (get/set zones)

**who said this was needed?** the wish explicitly requests "get/set operations to see what domains are already in cloudflare"

**evidence:** zones are the foundational container for dns records. without zone management, you cannot manage dns.

**what if we didn't do this?** users would have no way to onboard domains into cloudflare via code. defeats the entire purpose.

**scope check:** appropriate. CRUD operations cover all realistic use cases.

**simpler way?** no. this is already minimal — get, set, del.

**verdict: required**

---

## requirement 2: dns record management (get/set records)

**who said this was needed?** the wish explicitly requests "get/set operations to manage the dns of those domains"

**evidence:** dns records are the core value of dns. A records, CNAME, MX, TXT — this is what users actually configure.

**what if we didn't do this?** users could create zones but not add any records. useless.

**scope check:** appropriate. CRUD + zone reference.

**simpler way?** no. already minimal.

**verdict: required**

---

## requirement 3: domain registration management

**who said this was needed?** the wish requests "get/set operations to manage the import of domains into cloudflare, transfered out of another registrar"

**evidence:** registration settings (autoRenew, locked, privacyProtection) are real cloudflare registrar features.

**what if we didn't do this?** users could manage dns but not registration settings. partial solution.

**scope check:** appropriate. read + update. no delete (you don't "delete" a registration).

**simpler way?** could skip if only dns matters. but wish explicitly includes registration management.

**verdict: required per wish**

---

## requirement 4: transfer/purchase guidance via HelpfulError

**who said this was needed?** implicit from wish ("import domains into cloudflare"). cloudflare api doesn't support automated transfers.

**evidence:** cloudflare api docs confirm: registrar endpoints allow list and update operations, not purchase or transfer.

**what if we didn't do this?** users would hit an opaque error when they attempt to register/transfer. bad UX.

**scope check:** appropriate. guidance is scoped to what's needed — steps at both registrars.

**simpler way?** could just throw "not supported" error. but that leaves users stuck. HelpfulError is the pit-of-success approach.

**verdict: required for good UX**

## requirement 5: WHOIS lookup for transfer vs purchase detection

**who said this was needed?** derived from requirement 4. to give correct guidance, must know if domain is registered elsewhere or unregistered.

**evidence:** cloudflare provides `/intel/whois` endpoint. this is the only way to programmatically detect registrar.

**what if we didn't do this?** would have to show generic "transfer OR purchase" guidance. less helpful.

**scope check:** tight — single api call, single use case.

**simpler way?** could ask user to specify. but that's extra friction. auto-detection is better UX.

**verdict: nice-to-have, but strongly recommended**

## harder questions

### question: do we need three separate domain objects?

zone and registration both have `name` as the key. could they be one object?

**answer:** no. a zone can exist without registration (domain registered at godaddy, dns hosted at cloudflare). a registration can exist without a zone being actively used. they have independent lifecycles.

### question: is WHOIS really needed or excessive scope?

**answer:** without WHOIS, we show "domain not found — transfer or purchase required". with WHOIS, we show "domain registered at godaddy — transfer required" with godaddy-specific steps. the difference is significant for UX. promoting from "recommended" to **required**.

### question: is registrar-specific guidance excessive scope?

the vision shows links to godaddy, squarespace, route53 transfer docs. is this too much?

**answer:** these links are maintenance burden — they can go stale. **simpler approach:** show only cloudflare transfer docs + current registrar name. user can search "[registrar] transfer domain out" themselves.

**issue found:** registrar-specific transfer guides should be removed from vision scope. keep only cloudflare docs + registrar name from WHOIS.

### question: does the wish actually require del operations?

the wish says "get/set" not "get/set/del".

**answer:** del is implied for declarative control. if you can create but not delete, you can't clean up mistakes or offboard domains. del stays.

### question: is BadRequestError the right error type for guidance?

the vision uses `BadRequestError.throw()` for transfer/purchase guidance. but "bad request" implies user error. is "domain not in cloudflare registrar" really a bad request, or is it just an api limitation?

**answer:** examined more closely — the user DID request something invalid from the API's perspective (findsert a registration that doesn't exist). the error IS a bad request in the technical sense. BUT the guidance transforms it into actionable steps. this is correct usage. no change needed.

### question: is the dns record unique key correct?

the vision implies `(zone, name, type)` as the unique key. but dns allows multiple A records for the same name (round-robin). wouldn't that break uniqueness?

**answer:** examined cloudflare API — each record has its own `id`. the declastruct unique key should probably be `(zone, name, type, content)` or we need to reconsider how multiple records of same type are handled.

**potential issue found:** dns record unique key needs clarification. multiple A records for load balancing is a valid use case. current vision doesn't address this.

**resolution:** the vision should clarify how multiple records of same name+type are handled. options:
1. use `(zone, name, type, content)` as unique key
2. use cloudflare's record `id` as primary, with `(zone, name, type)` as partial key
3. explicitly scope out multi-record scenarios for v1

this is a **scope clarification needed**, not a blocker for vision approval.

### question: what happens with proxied on non-proxyable records?

the vision shows `proxied: true` on A and CNAME records, but not on MX. cloudflare doesn't proxy email traffic. what if someone sets `proxied: true` on MX?

**answer:** cloudflare API rejects this — error bubbles up as shown in edge cases table. the vision handles this correctly via "cloudflare api rejects; error bubbles up". no change needed.

### question: is setDomainRegistration with upsert meaningful?

vision shows `findsert` throws guidance when domain not in cloudflare. what about `upsert`? can you "upsert" a registration for a domain you don't own?

**answer:** no — upsert implies "update if exists, create if not". but you can't "create" a registration via API (requires manual transfer/purchase). so `upsert` on a non-existent registration should also throw guidance.

**issue found:** the vision's usecases table shows `setDomainRegistration({ upsert })` for "update domain settings" — but doesn't clarify that upsert on non-existent domain should also throw guidance just like findsert.

**resolution needed:** clarify in vision that both findsert and upsert throw guidance when domain not in cloudflare registrar.

### question: does the vision actually fulfill the wish for transfer operations?

the wish explicitly says "get/set operations to manage the import of domains into cloudflare, transfered out of another registrar"

the vision provides: HelpfulError guidance, NOT actual get/set operations.

**is this a mismatch?**

**analysis:**
- the wish says "get/set operations" — which implies programmatic control
- the vision delivers "guidance" — manual steps with helpful instructions
- technically, this is NOT what the wish asked for

**however:**
- the cloudflare API genuinely does not support automated transfers
- the wish cannot be literally fulfilled given API limitations
- the vision explicitly acknowledges this limitation in "what's awkward" section
- the HelpfulError pattern is the best possible UX given constraints

**should we change the wish?**

the wish could be updated to reflect reality:
```
- get/set operations to manage the import of domains into cloudflare, transfered out of another registrar
  (note: cloudflare API does not support automated transfers; declastruct provides step-by-step guidance instead)
```

**verdict:** the vision is honest about limitations but the wish may set incorrect expectations. this is a documentation concern, not a vision defect. the wish should be annotated at implementation phase to clarify API limitations.

**action:** no change to vision required. recommend annotate wish at implementation phase.

## issues found

### issue 1: registrar-specific transfer guides (reviewed)

**initial concern:** registrar-specific transfer guides (godaddy, squarespace, route53 links) are maintenance burden and can go stale.

**human feedback:** broken link is the same as no link. helpfulness outweighs maintenance burden.

**decision:** keep registrar-specific links. show both cloudflare docs and registrar-specific transfer-out docs when we recognize the registrar.

**vision includes:**
- cloudflare transfer docs (always)
- registrar-specific docs when recognized (godaddy, route53, namecheap, etc.)

### issue 2: upsert registration guidance unclear (fixed)

the usecases table shows `setDomainRegistration({ upsert })` for "update domain settings" but doesn't clarify behavior when domain doesn't exist in cloudflare. both findsert and upsert should throw guidance for non-existent domains.

**action taken:** updated usecases table:
- clarified "update domain settings" only works when domain exists in cloudflare
- added findsert/upsert to register and transfer rows (both throw guidance)

### issue 3: dns record unique key ambiguity (fixed)

the vision implies `(zone, name, type)` as unique key but dns allows multiple records of same name+type (e.g., multiple A records for round-robin). needs clarification.

**action taken:** updated domain knowledge lesson #7 to use `(zone_id, name, type, content)` as unique key, explicitly noted multi-record support for load distribution.
