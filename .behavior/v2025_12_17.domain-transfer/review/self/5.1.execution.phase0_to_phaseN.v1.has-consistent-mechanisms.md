### validation pattern analysis

read `castIntoDeclaredCloudflareDomainZone.ts`:

```typescript
const ZONE_TYPES = ['full', 'partial', 'secondary'] as const;
type ZoneType = (typeof ZONE_TYPES)[number];

const validateZoneType = (type: string | undefined): ZoneType => {
  if (!type || !ZONE_TYPES.includes(type as ZoneType))
    throw new UnexpectedCodePathError('invalid zone type from API', { type });
  return type as ZoneType;
};
```

read `castIntoDeclaredCloudflareDomainDnsRecord.ts`:

```typescript
const DNS_RECORD_TYPES = ['A', 'AAAA', ...] as const;

const validateDnsRecordType = (type: string | undefined) => {
  if (!type || !DNS_RECORD_TYPES.includes(type as ...))
    throw new UnexpectedCodePathError('invalid DNS record type from API', { type });
  return type as DeclaredCloudflareDomainDnsRecordType;
};
```

**found**: same pattern across both files:
1. define const array of allowed values
2. local validate* function checks value
3. throw UnexpectedCodePathError if invalid
4. return typed value

**should these be a shared utility?**

no. each validation is domain-specific:
- zone types are cloudflare zone concepts
- dns record types are dns protocol concepts
- no shared abstraction would help

extracting a generic `validateEnumValue()` would add complexity without benefit per rule.prefer.wet-over-dry.
