# 5 whys review: 1.vision

## requirement 1: declarative zone management

**surface request**: get/set operations to see what domains are in cloudflare

### 5 whys

1. **why is this needed?**
   → because teams need to know what zones exist in their cloudflare account

2. **why do teams need to know what zones exist?**
   → because without visibility, they cannot manage dns or troubleshoot issues

3. **why can they not use the cloudflare dashboard for visibility?**
   → because dashboard is manual, not versioned, not reproducible across environments

4. **why does it need to be versioned and reproducible?**
   → because infrastructure drift causes outages and tribal knowledge creates bus factor risk

5. **why does drift cause outages?**
   → because when reality diverges from expectation, deployments break and debug work is blind

**root need**: eliminate infrastructure drift and tribal knowledge for dns

---

## requirement 2: declarative dns record management

**surface request**: get/set operations to manage dns records

### 5 whys

1. **why is this needed?**
   → because teams add/update/delete dns records frequently

2. **why do teams need to manage dns records via code?**
   → because manual changes are error-prone and untracked

3. **why are manual changes error-prone?**
   → because humans make typos, forget to update all environments, and lack review process

4. **why does lack of review process matter?**
   → because dns errors (wrong IP, absent record) cause downtime with no paper trail

5. **why is paper trail important?**
   → because post-incident review needs to know who changed what, when, and why

**root need**: auditable, reviewable dns changes with human error prevention

---

## requirement 3: declarative domain registration

**surface request**: get/set operations to register new domains

### 5 whys

1. **why is this needed?**
   → because teams acquire new domains for new products/services

2. **why manage registration via code instead of dashboard?**
   → because registration options (auto-renew, lock, privacy) should be consistent

3. **why should registration options be consistent?**
   → because inconsistent options cause surprise domain expirations or security issues

4. **why do surprise expirations happen?**
   → because someone forgot to enable auto-renew, or the option was changed manually

5. **why not just set it once in dashboard?**
   → because dashboard changes are invisible; code review catches "auto-renew: false"

**root need**: prevent surprise domain expirations via visible, reviewable configuration

---

## requirement 4: domain transfer guidance

**surface request**: get/set operations to transfer domains into cloudflare

### 5 whys

1. **why is this needed?**
   → because teams migrate from other registrars to consolidate at cloudflare

2. **why consolidate at cloudflare?**
   → because multiple registrars = multiple dashboards = higher cognitive load

3. **why not just do transfers manually?**
   → because transfer process has 10+ steps, easy to forget one, takes 5 days

4. **why is a long multi-step process problematic?**
   → because context is lost between steps; "where was i?" becomes "start over"

5. **why does context loss matter?**
   → because repeated failures to transfer create frustration and wasted time

**root need**: reduce transfer friction by preserved context across multi-day process

## synthesis: what are the true root needs?

| requirement | root need |
|-------------|-----------|
| zone management | eliminate drift and tribal knowledge |
| dns records | auditable changes with error prevention |
| registration | prevent surprise expirations |
| transfer | reduce multi-day process friction |

### the deeper pattern

all four root needs point to one meta-need:

> **make domain infrastructure visible, reviewable, and resumable**

- visible: know what exists
- reviewable: changes go through PR
- resumable: pick up where you left off

### does the vision address the root needs?

| root need | vision addresses? |
|-----------|-------------------|
| eliminate drift | YES — `declastruct plan` shows drift |
| auditable changes | YES — resources.ts in git |
| prevent expirations | YES — registration options in code |
| reduce transfer friction | YES — HelpfulError preserves context |

### articulation: what i found

**requirements 1-3 (zones, dns records, registration options)**: the 5 whys confirm these are well-aligned. root needs match vision solutions. no gaps.

**requirement 4 (domain transfer)**: the 5 whys revealed a subtle tension.

the user's surface request: "get/set operations to transfer domains"
the vision's solution: HelpfulError guidance (not get/set)

this is a compromise, not a full solution. the root need is "reduce friction via preserved context." but does HelpfulError actually preserve context?

the vision claims:
> "`transferIn` status fields track progress — guidance can resume where you left off"

for this to work, declastruct must:
1. detect transfer-in-progress via cloudflare api
2. provide stage-appropriate guidance (not repeat step 1 when you're at step 7)
3. eventually succeed when transfer completes

**the gap**: the vision does not fully articulate what happens during the 5-day transfer window. what error message at each stage? when does "guidance" become "success"?

**is this gap a blocker?**

no. the vision correctly identifies:
- the root need (reduce friction)
- the approach (status-aware guidance)
- the mechanism (transferIn fields)

the exact behavior at each transfer stage is blueprint detail, not vision scope.

**one issue to flag**: the vision says `setDomainRegistration({ findsert })` is used for transfer. but findsert semantics imply "find extant or insert new." neither applies to transfer. transfer is "import from elsewhere" — a third mode.

this may warrant a design decision in blueprint:
- option A: findsert handles three cases (find extant, insert new, import transfer)
- option B: add `importTransfer` as distinct operation
- option C: keep findsert, but document that "insert" includes import-via-transfer
